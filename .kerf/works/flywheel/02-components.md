## ⚠️ THE PRODUCT FORK (must resolve before change-design — surfaced by the architecture review)
Research converged on **Architecture B** (deterministic code loop + LLM as a called-function, woken only for ~3 judgment classes). The architecture reviewer's sharpest critique: **B may answer a different question than the user asked.** The user asked to "keep an *agent* going indefinitely"; B delivers a *smarter daemon* that consults an LLM rarely. They are different products:
- **(B) Smarter daemon:** mostly Go; LLM woken on exceptions + empty-queue. Cheap, robust, debuggable. CANNOT (without extension) notice a class of beads failing systemically, observe throughput decline and hypothesize, or re-strategize priorities from reviewer-verdict patterns — i.e. the *judgment* the human orchestrator does today.
- **(A-fixed) Persistent LLM loop done right:** the orchestrator stays an LLM loop (which already works), and flywheel fixes its two actual defects — context exhaustion (regime-B fresh-context + digest) and crash-recovery (watermark + reconciliation). Keeps the "agent that thinks/adapts" behavior; costs more tokens.
These aren't mutually exclusive — **the likely answer is "B's deterministic skeleton + a widened LLM judgment surface"**: add a `pattern_detected` wake (N failures in an area, BLOCK-rate over threshold, throughput decline) fed a rolling outcome summary, and an LLM-emittable `set_max_concurrent`. **This is the #1 decision for the user.** (See change-design + open-questions.)

## A. The one new spec area

### `specs/cognition-loop.md` (NEW — working name; aka "the flywheel")
The orchestrator/cognition loop as a specified long-running process — the layer above the deterministic daemon work-loop. Today unspecified: PL-019 names the orchestrator-agent only to box it "OPTIONAL/post-MVH"; the daemon is LLM-free by **PL-018** (not PL-020 — correction). Must describe:
- **CL-architecture** — Architecture B + the widened judgment surface (resolve the fork); single-vs-multi rationale + named scale-trigger to revisit (sustained max_concurrent≥5 w/ frequent failures, or stateful cross-cycle investigations).
- **CL-cycle** — sense (reconcile from durable artifacts) → consume events (wake-filter) → act (dispatch/react/investigate/backfill) → persist (notes+watermark) → continue-or-recycle → loop. Plus the **crash/restart sequence** (steal Symphony §14.3: restart = reconstruct in-flight from durable state, adopt orphaned runs, bootstrap a fresh session).
- **CL-context** — regime-B: fresh context per recycle; byte-stable cacheable prefix + small post-breakpoint digest; recycle trigger; cache-stability runtime invariant (`cache_read ≈ prefix_size`, alert on drop). Engage-or-bound the small-recent-turns-buffer alternative (Hermes-LCM fresh-tail) rather than ignore it.
- **CL-digest** — computed view over an append-only log (derivable stream + `notes.jsonl`); cut-rule; ≤N-line budget; resolution-marking (auto-resolve restricted to `defer` notes; `hypothesis`/`warning` need explicit resolve); **coercive note discipline** via daemon-emitted `decision_required` (unacknowledged → a digest exception that blocks dispatch); the `harmonik digest` projector (schema-versioned, unknown-event→warn) + `--health` metric.
- **CL-substrate** — runtime decision (see §C-1 — raw Anthropic Messages API on an API key; NOT the Pi-OAuth-Max path, which is ToS-banned) + billing reality + model-version-pin vs cache stability.
- **CL-consistency** — read-only sensing pass; authority lattice with **origin/main two-phase done** (§C-3 correction); watermark (crc+schema-version, never-regress, replay-from-floor-on-corruption) + reacted_ledger (fsync before watermark advance); per-reaction-class idempotency (not one label); loop-singleton lock; effectively-once.
- **CL-queue-pressure** — WIP-limited pull; ceiling==target==max_concurrent (structural over-aggression guard) + optional LLM `set_max_concurrent` override; eager pure-code refill; drain-bias; no-speculative-work guard; metrics (`refill_misses` headline); batch-formation oracle (is it `kerf next` top-N, or top-N + conflict/area-cap filter? — must specify).
- **CL-operator** — start/stop/pause/inspect; tmux-inspectability carried to the substrate (locked C9); cost-ceiling / kill-switch; "what does "the loop is wrong" look like and how is it caught" (digest-health + a watch surface).

## C. Cross-component decisions (carry to change-design / user) — with verification corrections

1. **[CORRECTION+DECISION — substrate] Pi-via-Max-OAuth is DO-NOT-SHIP.** Verified: Anthropic ToS update (Feb 2026) + enforcement (eff. 2026-04-04) bans reverse-engineered Claude-Code OAuth in third-party tools (Pi's class; "stealth mode" comments = evidence). Pi-as-code is fine; the OAuth-for-Max path is not. → **Primary substrate = raw Anthropic Messages API on an API key** (own thin loop, or `pi-agent-core` driven with an API key). Cache cost model survives (API pricing identical). **Reframes G8: the "exception" is accepting API per-token billing for headless use** (Max isn't available for unattended loops; the workaround is banned). USER decision: own-loop vs pi-agent-core-w/-API-key; accept per-token cost.
2. **[CORRECTION — queue] No new queue kind needed (likely).** Verified: stream groups skip dispatched heads → already append-able AND concurrent at max_concurrent>1. The research's "need `--wave` for concurrency / build a `pool` kind" premise is FALSE. Simplifies the queue change to ~zero. Confirm in change-design.
3. **[CORRECTION — consistency] "git wins" = `origin/main`, two-phase done.** The daemon terminal sequence (local update-ref → push → reset → outcome_emitted → CloseBead → bead_closed → run_completed) is multi-step non-atomic; an external observer reading LOCAL main mid-sequence sees false-done and can double-react if push then fails (EM-053 reopens). FIX: react-as-complete only when `run_completed{success}` event AND trailer on `origin/main`.
4. **[DECISION — note discipline] coercive via `decision_required`** (daemon emits on defer/block/failed-twice/merge-conflict; unacknowledged = blocking digest exception). Adopt (review-strong).
5. **[DECISION — process placement] separate process vs harmonik-workflow.** Working assumption: separate process (+ singleton lock). If a harmonik workflow, gets checkpoints/replay free. USER/design.
6. **[CORRECTION — hk-24xn1 CLOSED]** Daemon already wakes on queue-submit/append (`SetQueue`→`wakeC`, 2026-05-26). Eager refill has no blocking dependency. Remove "wake-on-append gap" from design text.
7. **[CORRECTION — event_id_hwm NOT implemented]** EV-002c (cross-restart UUIDv7 monotonicity via an HWM file) is spec-only + an audit sensor, NOT code. Watermark still safe (UUIDv7 ms-order + `ScanAfter` graceful-on-missing-file), but do NOT cite the HWM file as an existing primitive.
8. **[CORRECTION — ScanAfter(missingID)] not cold-start.** It yields only events > the parsed UUID; a garbage-but-parseable watermark silently skips the gap. → the loop MUST implement "corrupt/parse-fail → replay from JSONL floor using reacted_ledger as dedup authority." Design obligation, not substrate.
9. **[DECISION — widen judgment surface]** add `pattern_detected` wake + LLM `set_max_concurrent` override (architecture review) — resolves the product fork toward "B-skeleton + real judgment."
10. **[OPEN — verify] Gastown `respawn-pane --continue` fresh-vs-replay** (does --continue replay the transcript, contradicting "fresh context"? SDK verify confirms --continue replays full transcript → Gastown's "fresh" likely depends on clearing the session JSONL first; relevant to our recycle mechanism design).
