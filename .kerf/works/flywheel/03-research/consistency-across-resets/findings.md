# Research/Design — Consistency & reconciliation across context resets

> Component: `consistency-across-resets`. Source: research sub-agent (opus), grounded in actual harmonik code/specs, 2026-05-27. **[FACT]** = grounded in harmonik code; **[PROPOSAL]** = new design.

## TL;DR
- **The daemon already owns run-level crash-consistency** (atomic Beads claim *before* `run_started`; git-wins completion authority; 11-category reconciliation taxonomy; UUIDv7-monotonic event log with `ScanAfter`/watermark; idempotency-keyed commit/merge/close). The cognition loop must NOT re-implement any of it — it is a *second independent consumer* of the same durable substrate. Its only new obligation: **idempotent reaction with its own durable watermark.**
- **No-double-dispatch is free** from the daemon's atomic claim + queue.json status — the loop reconciles desired-vs-observed (Kubernetes-style), never blind re-issues.
- **No-dropped-completion / no-missed-failure is a watermark-replay problem:** loop persists "processed through event_id N" to `.harmonik/cognition/watermark.json`; on resume `ScanAfter(N)` replays only the gap; reactions made idempotent (keyed on event_id) → effectively-once.

## Division of labor (Q7 — everything depends on it)
DAEMON already guarantees [FACT]: atomic Beads claim persists before `run_started` (`internal/core/runstartedpayload.go:9`); `queue.json` mirrors observed state (item `status ∈ {pending,dispatched,completed,failed,...}` + minted `run_id`, `queue-model.md §2.7`); git is completion authority (`Refs:`/`Harmonik-Run-ID` merged to main = done, RC-INV-001, execution-model §4.7); commit/merge/close via idempotency-keyed adapter (`<run_id>:<transition_id>:<op>`, `internal/core/idempotencykey.go`); 11-category reconciliation taxonomy reconstructs in-flight truth from git+Beads at daemon startup with per-run `flock` (RC-002a).
COGNITION LOOP must additionally guarantee [PROPOSAL]: the daemon reconciles *runs*; the loop reconciles *its own reactions to run outcomes* ("on run_failed, file follow-up"; "on REQUEST_CHANGES, queue a fix") — these have NO daemon-side idempotency, so the loop is the at-risk layer. Contract: (1) never submit a bead the daemon already shows claimed; (2) process every terminal event effectively-once via watermark + dedup ledger.
**Do not duplicate** (honors hk-hqwn.31 / RC-014): loop must NOT walk JSONL to reconstruct run state, must NOT re-derive completion from anything but git, must NOT take reconciliation locks. It reads queue.json + `br` + git for *truth*, events only for *what-changed-since-N*.

## Reconciliation on resume (Q1) — read-only sensing pass, by authority order
1. Read watermark `.harmonik/cognition/watermark.json` → `last_processed_event_id N` + `reacted_ledger`. Missing/corrupt → cold start (replay from earliest available; expensive but safe).
2. Read `queue.json` → observed dispatch state (which beads pending/dispatched(+run_id)/completed/failed).
3. Git completion truth: for each dispatched run_id, `git log --all --grep "Harmonik-Run-ID: <run_id>"` on main → merged checkpoint landed? **Git wins** (RC-INV-001).
4. `br show` each bead → corroborate closed/in_progress.
5. `ScanAfter(N)` on events.jsonl [FACT `internal/eventbus/jsonlwriter.go:311`] → replay only the gap; classify each terminal event against `reacted_ledger`.
**Authority lattice:** git(merged) > queue.json > Beads coarse > events.jsonl. If sources disagree unresolvably → it's a daemon reconciliation case; loop defers, acts only on cleanly-resolved outcomes.

## Idempotent dispatch (Q2) — check-observed-before-desired guard
Before `harmonik run --beads X`, for each X: already in queue.json {pending,dispatched,completed}? → skip. Git has merged Harmonik-Run-ID for X? → skip + close stale bead (the CLAUDE.md pre-screen). `run_started` with no terminal & no merge? → skip (in-flight). Only if in none → submit. Submit key = bead-id within an open queue; daemon refuses to double-claim an `in_progress` bead (Beads atomic write). The kerf "atomic claim file, no lock daemon" idea is ALREADY realized as the Beads atomic write + `flock` reconciliation locks — reuse, don't reinvent. run_id is per-dispatch [FACT queue-model §2.7]; retries = new run_ids (execution-model EM-014); "double-dispatch" only means submitting while a prior run is in-flight (guard prevents).

## Watermark/checkpoint (Q3)
`.harmonik/cognition/watermark.json`: `{schema_version, last_processed_event_id:<UUIDv7>, reacted_ledger:{<event_id>:<reaction_ref>}}`. Use UUIDv7 (strictly monotonic within process + across restart via the existing `event_id_hwm` file, EV-002a/c), NOT a byte offset (breaks on rotation). `ScanAfter(last_processed)` reads only new lines each tick → **the token-cost answer** (never read the whole log). Atomic write (temp+rename, WM-026 pattern). **Advance watermark AFTER recording the reaction.** The loop's digest is *derived* from queue.json+git+gap, never authoritative; the watermark bounds how much log feeds the digest.
**[VERIFY harmonik-internal]** Is `ScanAfter(missingID)` graceful (replay-from-start) or error? Affects cold-start + compaction handling.

## Effectively-once reaction (Q4)
Exactly-once *delivery* across a crash is impossible (two-generals). Target = at-least-once + idempotent effect = effectively-once (outbox/event-sourcing consensus). Pattern: handlers keyed on triggering event_id; check `reacted_ledger[event_id]` (skip if present); make effects idempotent — "file follow-up on run_failed" → deterministic dedup label `reaction:<event_id>` on the created bead, `br list --label reaction:<event_id>` before create. **Record-then-advance:** write ledger entry (atomic) THEN advance watermark. Crash between effect & ledger → dedup label makes retry a no-op; crash between ledger & watermark → re-read finds it in ledger, skips.

## The 10-in-flight scenario (Q5 — ACCEPTANCE TEST)
Submit A..J → queue.json 10 items, all `dispatched` (run_ids R_A..R_J), watermark at N₀. Context recycles. During gap: C,F,I complete (merged checkpoints, closed, `run_completed`); D fails (`run_failed`, no merge); A,B,E,G,H,J running. Fresh boot runs the sensing pass: watermark=N₀; queue.json shows C/F/I=completed, D=failed, rest=dispatched; git confirms C/F/I merged + D not-landed; `br show` confirms; `ScanAfter(N₀)` replays run_completed{C,F,I}+run_failed{D}, none in ledger. Actions: react to C/F/I completions (advance kerf baseline/log), ledger+watermark; react to D (file follow-up labeled reaction:<event_id_D>, idempotent), ledger+watermark; A,B,E,G,H,J → do NOT re-dispatch (guard: they're `dispatched`), resume monitoring. **Result: zero double-dispatch, zero dropped completion, zero missed failure.** ✅ Each artifact's role: queue.json=fast observed mirror; git=completion ground truth; events gap=what-changed delta; watermark=replay only N₀→now; ledger=no double-react on re-crash.

## Crash windows (Q6) — all survive
after-submit-before-queue-item (daemon writes atomically; either exists→skip or not→safe re-submit); after-claim-before-loop-records (loop holds no claim state; reads run_started as in-flight, guard prevents double); after-complete-before-react (watermark still at pre-completion N; ScanAfter replays); mid-reaction (dedup key → no-op); ledger-written-watermark-not (re-read hits ledger → skip); watermark-advanced-past-unreacted (MUST NOT happen — enforced ordering effect→ledger→watermark is the single load-bearing invariant).
