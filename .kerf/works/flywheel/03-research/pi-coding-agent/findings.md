# Research — Pi coding-agent as the loop-runtime substrate

> Component: `pi-coding-agent`. Source: research sub-agent (opus) over github.com/earendil-works/pi, 2026-05-27. **Verdict: VIABLE — strongest candidate substrate.** Build on `@earendil-works/pi-agent-core` (the harness/loop), NOT `pi-coding-agent` (the CLI). MIT, TypeScript (Node+Bun), actively maintained (v0.76.0, same-day commits, 56k stars). Pin a version (pre-1.0 churn).

## TL;DR
- **`pi-agent-core` is a clean, hackable, headless agent loop** with first-class hooks (`transformContext`, `shouldStopAfterTurn`, `convertToLlm`, `prepareNextTurn`) that let us **own context construction every turn** — "fixed prefix + small digest, no compaction" drops in with near-zero framework-fighting. **Compaction is opt-in and lives only in the CLI layer, not the core loop** — if we build on `pi-agent-core` directly, *there is no compaction to disable.*
- **The Claude Pro/Max OAuth path is implemented and working** — Pi authenticates exactly like Claude Code (same `claude.ai/oauth/authorize` client_id, PKCE, `user:sessions:claude_code` scope, `claude-code-20250219`+`oauth-2025-04-20` beta headers, `claude-cli` user-agent). **Decisive: we can keep Max billing without shelling out to official Claude Code** — but it's an *unofficial* path (mimics Claude Code identity; Anthropic could restrict it). ToS posture is the user's call.
- **Prompt caching is real and correct** — ephemeral `cache_control` breakpoints on system + last tool + trailing message (the recommended incremental pattern); 1-hour TTL available. The framework does NOT mutate the prefix; prefix byte-stability is *our* responsibility via `transformContext`, and Pi won't fight us.

## 1. Architecture — the loop
`runLoop()` in `packages/agent/src/agent-loop.ts:155-269`. Two-level: outer polls follow-ups; inner runs while `hasMoreToolCalls || pendingMessages>0`. Each turn: inject pending/steering msgs (`:182-190`) → stream assistant (`:193`) → filter tool calls (`:203`) → execute (parallel default, `:208`) → push results into `currentContext.messages` (`:212-215`) → `turn_end` → `prepareNextTurn` (`:226`) → `shouldStopAfterTurn` (`:242`). Model boundary = `streamAssistantResponse()` (`:275-368`): applies `transformContext` (`:284`) → `convertToLlm` (`:289`) → builds `Context={systemPrompt,messages,tools}` (`:292`) → provider `streamFn`. **History is a plain `AgentMessage[]` we hand in each turn — no hidden state.** The `AgentHarness` (`harness/agent-harness.ts`) persists each message to an append-only session tree and **rebuilds turn-state from the session every turn** (`createTurnState()`→`session.buildContext()`, `:313-345`, via `prepareNextTurn` `:439-448`).

## 2. Context control — full, by design
`AgentLoopConfig.transformContext` (`types.ts:186`) is documented verbatim for "Context window management (pruning old messages)" and "Injecting context from external sources." It receives the message array and returns a replacement right before `convertToLlm` — **exactly where we inject "fixed prefix + deterministic digest" and drop everything else. No sliding window is imposed; we decide the message list.** The *loop* never auto-compacts. `harness.compact()` (`agent-harness.ts:681`) only runs if the host calls it, and a `session_before_compact` hook can cancel. The auto-compaction *trigger* (`shouldCompact`, `setAutoCompactionEnabled(false)` at `agent-session.ts:2032`) lives only in the CLI package — absent from `pi-agent-core`.

## 3. Prompt caching — decisive-positive
`packages/ai/src/providers/anthropic.ts`: `getCacheControl()` (`:54-67`) builds ephemeral breakpoint (`ttl:"1h"` optional). `buildParams()` puts `cache_control` on system block(s) (`:909,916,925`), last tool def (`convertTools`, `:1187`), and the **last** user/tool-result block (`convertMessages`, `:1137-1158`) = trailing-message incremental pattern Anthropic recommends. Usage telemetry tracks `cacheRead`/`cacheWrite` (`:531-532,669-673`). **Framework does not mutate the prefix** — order is whatever array we pass; only the tail gets a cache marker. So our stable prefix survives *iff our `transformContext` emits a byte-identical prefix each cycle* (in our control). `CacheRetention=none|short|long` (`PI_CACHE_RETENTION=long`→1h).

## 4. Model/provider + billing — the headline
Providers (`register-builtins.ts`): Anthropic, OpenAI (completions/responses/Codex), Google (GenAI/Vertex), Bedrock, Mistral, Azure, Copilot, Cloudflare, OpenRouter, Fireworks, z.ai. Anthropic via `@anthropic-ai/sdk@0.91.1`. **Billing is auth-mode-dependent, not hardcoded:** `createClient()` (`anthropic.ts:779-887`) branches on token shape — if key matches `sk-ant-oat...` (`isOAuthToken`, `:775`) it uses **Bearer auth + Claude Code identity headers and forces the system prompt to begin "You are Claude Code, Anthropic's official CLI for Claude."** (`:904-918`) → routes through the **Claude Pro/Max subscription, NOT API-key billing.** Full OAuth flow built (`utils/oauth/anthropic.ts`): provider named "Anthropic (Claude Pro/Max)" (`:383`), PKCE, `user:sessions:claude_code` scope, refresh rotation (`:348`). **Max billing is NOT impossible outside official Claude Code — Pi reproduces the dance.** HONEST CAVEAT: relies on mimicking Claude Code's client identity (comment `:69` "Stealth mode: Mimic Claude Code's tool naming exactly", `claudeCodeVersion="2.1.75"`). Works today; unofficial; Anthropic could break it; ToS judgment is the user's. API-key billing is the fallback mode.

## 5. Persistence / resume — yes
Sessions = append-only JSONL tree (`harness/session/jsonl-storage.ts`): header line + one entry/line (`{id,parentId,timestamp,payload}` where payload ∈ message|model_change|compaction|branch_summary|**custom**|label|leaf). `buildSessionContext()` (`session.ts:21-76`) reconstructs messages by walking leaf→root. **A fresh process re-opens the JSONL and replays it → full crash resume, incl. active model/thinking-level.** NOT persisted: in-memory steering/follow-up queues. For "survive crash with in-flight work tracked," persist our own digest/work-ledger as `custom` entries (`appendCustomEntry`, `session.ts:179`).

## 6. Extensibility
TS on Node+Bun; **MIT** all packages. `pi-agent-core` is thin/standalone (deps: only `pi-ai`,`ignore`,`typebox`,`yaml`) — depend directly, ignore CLI/TUI. Loop ~740 LOC, harness ~1000, readable. Hook surface covers everything: **indefinite run** = never return true from `shouldStopAfterTurn` until our recycle trigger; **recycle-near-limit** = return true (documented "before context gets too full"), persist digest, start fresh loop; **digest injection** = `transformContext`; **event reaction** = inject via steer/followUp/nextTurn queues or a custom tool. Forking effort LOW — mostly configuring existing hooks. Risk: high release velocity (pin version).

## 7. Pi vs Claude Code SDK — comparison axes & honest read
(a) **Max billing** — Pi: yes via reproduced OAuth, *unofficial*; SDK/`claude -p`: yes but official surface (separate credit pool, see sdk findings). (b) **Context ownership** — Pi: total (we build every turn's array); SDK: opaque (feeds prompts, manages context/compaction internally) — *the core reason Pi wins for no-compaction+digest.* (c) **Cache-prefix control** — Pi: we own prefix + breakpoints; SDK: managed, less control. (d) **Recycle policy** — Pi: `shouldStopAfterTurn`+fresh-loop (ours); SDK: tied to its session/compaction lifecycle. (e) **Persistence** — both persist; Pi's JSONL is open & extensible (custom entries). (f) **Hooks** — Pi: rich in-process; SDK: process-boundary. (g) **Legitimacy/stability** — SDK wins (official); Pi third-party, must track identity-header changes.
**Honest read: Pi wins decisively on the two things that define this project — total context construction control and explicit caching — matches on Max billing (unofficial caveat). SDK abstracts away exactly the control we need.**

## Recommendation
Depend on `pi-agent-core` + `pi-ai` (skip CLI/TUI); drive `runAgentLoop`/`AgentHarness`; implement the loop via `transformContext` (digest), `shouldStopAfterTurn` (recycle), `custom` session entries (work ledger); auth via the Anthropic OAuth provider for Max billing (with ToS/unofficial caveat noted); pin the Pi version. **Open decision for user: accept the unofficial Max-OAuth path (cheap, ToS-grey) vs. official Agent-SDK credit pool (legit, separate $100-200/mo budget) vs. API-key per-token.**
