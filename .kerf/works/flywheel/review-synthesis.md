## 2. Design-shaping critiques (the design must answer these)

**P1 — [PRODUCT FORK, highest] The corpus inverted the user's premise without flagging it.** User asked for "one or more *agents* in a loop"; 11/16 threads concluded "the loop is deterministic Go code, LLM is a called function ~3×/batch" (Architecture B). That's a *smarter daemon*, not an agent-loop — a scope reversal that needs the user's explicit sign-off. B can't (un-extended) notice a class of beads failing systemically, see throughput decline & hypothesize, or re-strategize from reviewer-verdict patterns — the judgment the human orchestrator does today. **Resolution proposed:** B's deterministic skeleton + a *widened judgment surface* (a `pattern_detected` wake fed a rolling outcome summary; an LLM-emittable `set_max_concurrent`). **User must ratify the framing.**

**P2 — [SELF-UNDERCUT] If Arch B calls the LLM ~3×/batch with a fresh small context, the context-window problem largely evaporates** — so much of the elaborate cache/digest/watermark machinery may be solving a problem B already dissolved. Conversely, the notes.jsonl design assumes a *persistent cognitive agent* that holds intent, but B says the LLM is stateless-per-call ("nothing accumulates") — **who writes the intent notes, and when? (contradiction C6).** The digest+notes story and Architecture B don't fully compose. change-design must reconcile: either the LLM cycle is long-lived-enough to accrue/append notes, or notes are emitted by the deterministic layer at decision points (the `decision_required` mechanism leans this way).

**P3 — [OVER-ANCHORING] "fresh context + store-is-memory" was treated as proven; two load-bearing mechanics are unverified/contradicted.** (a) Gastown `respawn --continue` "fresh context" vs SDK "`--continue` replays full transcript" (C1) — Gastown's "fresh" likely depends on clearing the session JSONL; **our recycle mechanism must explicitly start a NEW context, not `--continue`.** (b) The cache-reuse crux was sold as CONFIRMED in a TL;DR but flagged REASONED in the body (C2) — docs do support it (V7) but **measure it before committing.** The corpus risks ratifying Greg's own prior ("30-40 lines + fetch + fresh-digest") rather than challenging it. Note: compaction was rejected on quoted vibes + a Factory score that's for *free-form summaries*, NOT Anthropic's structured compaction — the strongest disconfirming experiment (run a native-compaction orchestrator 12h, measure what breaks) was never run.

**P4 — [OVER-SCOPED v1] The full stack (wake-filter tiers, 2s debounce, circuit-breaker, pool-queue, keep-warm pings, notes GC, tiered-TTL demotion) is speculative tuning of a loop nobody has watched run** — violates "don't add abstraction the user hasn't asked for." The daemon already owns claim-atomicity, git-completion-authority, idempotency keys, 11-category reconciliation; the loop's genuinely-new surface is much smaller (authoritative in-flight-run tracking across a reset).

**P5 — [DIGEST robustness] note-discipline is fragile (voluntary `note add` will silently drop intent on the hardest decisions) → make it coercive** via daemon `decision_required` events that become blocking digest exceptions. The **projector is a drift-prone SPOF** → version it, warn on unknown event types, add `harmonik digest --health` (note-filing rate, unknown-event coverage, open-note age). **Auto-resolve only `defer` notes**; `hypothesis`/`warning` need explicit resolve (a force-closed bead can kill a still-valid hypothesis).

**P6 — [CONSISTENCY holes] per-reaction idempotency, not one label.** `reaction:<event_id>` only makes create-bead idempotent; re-dispatch / `kerf --ack` / escalate-to-user are NOT naturally idempotent. Need a per-class strategy. **Missing loop-singleton lock** (`.harmonik/cognition/loop.lock`, flock+pid) — two accidental instances both react. **Never regress the watermark** (`max(persisted, heartbeat.last_event_id)`). **git-done-but-no-terminal-event after K heartbeats → Tier-2 wake** (daemon-crashed-mid-terminal), not silent advance.

## 3. Consolidated recommendation (minimal slice + the gates) — for change-design

**Build Ralph + the one missing piece, defer the rest until a loop has run.** The skeptic + architecture reviews converge:

**Minimal first slice (don't spec past step 3 until step 3 has run a day):**
1. **`harmonik digest` projector** — pure Go; reads queue.json + git(`origin/main`) + `ScanAfter(watermark)` + beads; emits the ≤40-line terse digest (worked example in context-mgmt thread Q6). Testable, deterministic, useful even to a human today. De-risks everything.
2. **Watermark + idempotent-reaction guard** — `.harmonik/cognition/watermark.json` (crc+schema-version, never-regress) + reacted_ledger (fsync before watermark advance) + per-reaction idempotency + loop-singleton lock. The **10-in-flight scenario = a real scenario-harness test, not prose.**
3. **The dumbest loop** — `pi-agent-core` (API key) driven by `transformContext` = `[frozen prefix | cache_control | harmonik digest]`, `shouldStopAfterTurn` recycle near a token threshold, **fresh context (NOT --continue)**, poll `harmonik subscribe`, react simply. Run unattended for hours against the real daemon; watch what breaks. Measure `cache_read_input_tokens` (verify V7), test double-dispatch avoidance, test the substrate over a multi-hour run.

**Gates that MUST exist before anything pushes unattended (currently zero coverage — the scariest gap):**
- **G-test: a replay test harness** — replay a recorded events.jsonl + queue.json through the loop and assert correct reactions WITHOUT tokens/real-push. You cannot trust an unattended pushing agent you can't deterministically replay-test.
- **G-liveness: a progress/liveness invariant + kill condition** for the cognition loop (analogue of the daemon's bounded-retry "no iteration without state change"); define what "the loop is wrong" looks like (doom-loop re-dispatch, projector hides a failed bead, wake-filter drops a `merge_conflict`) and how it's caught.
- **G-security: a threat model.** This is an LLM that commits/merges/pushes with no human gate for days, reading bead bodies / diffs / reviewer output it did NOT author — a prompt-injection surface ("ignore rules, force-push / dump env"). HIGHEST-severity gap; the corpus had zero coverage. Needs: input sanitization/trust boundaries, allowed-action allowlist, no-force-push guard, and the kill-switch below.
- **G-cost: a dollar/token ceiling + hard kill-switch.** The circuit-breaker (N reactions/window) limits dispatch rate, not spend. Need a per-day budget cap that halts the loop.
- **G-inspect: preserve tmux-inspectability (locked C9).** A headless Pi loop drops the tmux pane — the locked inspection affordance. Resolve: run the loop inside a tmux pane / provide an attach surface / a live `harmonik digest --watch`.

**Defer until observed:** wake-filter 3-tier taxonomy specifics, 2s debounce + circuit-breaker constants, keep-warm-ping cadence, notes GC / tiered-TTL-demotion, `set_max_concurrent` dynamics, the multi-role fleet (Architecture C). All are tuning of an unobserved loop.
