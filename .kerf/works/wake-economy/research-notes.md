## R1 — Scheduling primitive (O3/O4) [agent a534545, DONE]
**harmonik ALREADY has a config-driven scheduled-task primitive — reuse it.**
- CLI: `harmonik schedule add|list|remove|enable|disable|run-now`. Store: `.harmonik/schedules.json` (durable, survives restart). cmd/harmonik/schedule.go.
- Spec syntax: `"every@<dur>"` (e.g. 5m,1h) | `"daily@HH:MM [tz]"`. parseScheduleSpec schedule.go:107-141.
- Action kinds: `spawn-crew` (--crew/--queue/--mission) | `command` (-- argv). types.go:25-52.
- Policies: --overlap-policy skip|allow; --catchup off. overlapBlocks (command=Kill(pid,0); spawn-crew=comms who) scheduletick.go:162-191.
- Daemon integration: runScheduleTick once per 2s workloop poll (scheduletick.go:85-116); schedule.Decide() pure/testable; store.WakeCh() wakes idle loop on mutation; ReloadIfChanged picks up file edits.
- PRECEDENTS (auto-registered jobs): ops-monitor `every@5m` action=command `bash scripts/ops-monitor-check.sh` (opsmonitor_schedule.go:32-47, ensureOpsMonitorSchedule); ctx-watchdog `every@5m` (ctx_watchdog_schedule.go:41-56).
- Intervals + targets 100% config-driven (operator edits schedules.json or CLI). Matches no-hardcoded mandate. Keeper config parallel: .harmonik/config.yaml keeper: block (poll_interval, etc.).
- **GAP (genuine):** action kinds are only spawn-crew + command. There is NO `comms-send` action kind. If O3's "scheduled message that must take place" means POST A COMMS MESSAGE to the captain/watch-officer (not spawn a crew / run a script), that's net-new code (add Action.Kind="comms-send" with --to/--topic/--message), OR wrap via action=command `bash -c 'harmonik comms send ...'`. DESIGN DECISION needed: native comms-send action vs command-wrapper.
- O4 mutual-liveness: schedule can fire every@1h fine; the CHECK LOGIC (what to ping, what to post back) is the script/crew's job, not a schedule gap.

## R2 — Event ledger + comms filtering (O1/O6) [agent a9b3288, DONE]
- EVENT BUS: internal/eventbus/eventbus.go (Emit/EmitWithRunID/Subscribe). Durable .harmonik/events/events.jsonl. ~79 event types (internal/core/eventtype.go). Envelope (internal/core/event.go): event_id(UUIDv7), schema_version, type, timestamp_wall(RFC3339), timestamp_mono_nsec, run_id?, state_id?, source_subsystem, payload(RawMessage). Durability classes F(fsync)/O(ordinary)/L(lossy).
- Relevant event types: run_started/completed/failed, bead_closed, epic_completed, agent_ready/completed/failed, agent_warning_silent_hang, decision_required+decision_acknowledged (§8.12 operator gate, F-class), daemon_started/ready/shutdown, session_keeper_warn/cycle_complete/hard_ceiling, governor_signal, metric, consumer_failed/bus_overflow/dead_letter_enqueued, agent_message, agent_presence.
- AgentMessagePayload (internal/core/agentcommspayloads_djqc9.go:57-122): From(req), To(req,"*"=broadcast), Topic(opt), Body(req), InReplyTo(opt event_id). Dedupe on event_id (EV-018). F-class (fsync before send returns).
- `comms log` (read-only, NO daemon, scans events.jsonl via eventbus.ScanAfter): --since EVENT_ID|DUR, --to, --from, --topic, --json. GREAT for watch-officer ledger reads.
- `comms recv`: --agent, --from, --topic, --follow (subscribe tail, auto-reconnect backoff 1→10s), --wait, --timeout (exit 3), --json. Per-agent durable cursor (hk-tafd4) — do NOT contaminate by reusing for watch-officer.
- `comms send`: --to|--broadcast, --from(=$HARMONIK_AGENT), --topic, --reply-to, --wake, --socket, --project. Returns event_id.
- `comms who`: presence projection, ~120s TTL staleness, --json. `join`/`leave`: agent_presence beats.
- `subscribe` (cmd/harmonik/subscribe.go + internal/daemon/subscribe.go): --types t1,t2 (default all), --heartbeat(10-600s,def60), --since-event-id (replay cursor), --to/--from/--topic (agent_message addressing only), --json NDJSON of core.Event + {heartbeat}+{subscription_gap dropped:N}. Backpressure: 256-slot buffered, drop-oldest, gap counter.
- Filter predicate: MatchAgentMessage (internal/daemon/agent_message.go:33-44) — to==""||To∈{to,"*"}; from==""||From==from; topic==""||Topic==topic. Used by live + replay (N1).
- LEDGER RECOMMENDATION: reuse events.jsonl (ScanAfter) + a `.harmonik/watch-cursor` (last event_id) for watch-officer's own forward cursor; dedupe on event_id (N3). Don't reuse beads.db (terminal-state only) or the per-agent recv cursor.
- **GAPS (genuine, design must address):** (1) NO "operator/escalation channel" concept — only (from,to,topic). Need a CONVENTION (e.g. topic=escalation, or from=watch-officer + a reserved topic) for captain's filtered watcher. (2) `comms recv --from` is SINGULAR not repeatable → captain can't say "from {operator,watch-officer}" in one invocation → need repeatable --from OR a topic convention OR a new filter. (3) subscribe addressing filters apply to agent_message ONLY; other event types ignore --to/--from. (4) events.jsonl has no per-reader cursor (watch-officer maintains own).
