**What EM is about (2-3 sentence summary).** EM owns harmonik's run / checkpoint / outcome-spine data model: the eight core types (`Workflow`, `Node`, `Edge`, `Run`, `State`, `Transition`, `Checkpoint`, `Outcome`); the git-checkpoint-trail durability contract with structured trailers + transition-record sibling files at run-scoped paths; the durability decision procedure keyed on `transition_kind × outcome_status`; the orchestrator main loop, edge-selection cascade, sub-workflow expansion (with durable pin), and validation obligations; the six-class failure taxonomy (`transient` / `structural` / `deterministic` / `canceled` / `budget_exhausted` / `compilation_loop`); and the three-store authority rule (git wins on completion disagreement). It is normative for every subsystem that produces, consumes, or reasons about runs. EM depends on AR for the four-axis classification + ZFC tagging + role vocabulary; everything else is a forward co-reference resolved at corpus scale.
