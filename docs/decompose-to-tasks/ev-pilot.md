# EV Pilot — Decompose-to-Tasks Worked Example

`pilot-version: 0.1.1` — re-drafted 2026-04-28 against `specs/event-model.md` v0.3.3 (status `reviewed`) and discipline v0.8 (post EV r1 5-patch pass). Fourth instance of the decompose-to-tasks discipline applied to a normative spec; third behavioural-adjacent spec after BI and EM. EV is the canonical event-payload owner for the corpus; its §8 carries 78 typed event entries (the largest §8 in the corpus by an order of magnitude) and triggers discipline §2.11(c) per-row bead minting under a taxonomy umbrella. EV is taxonomy-first by spec-shape (not behavioural like EM or PL), so §4 reqs decompose flatly into single-declaration beads with no §2.2 multi-step splits; the dominant structural cost is the §8 event-row enumeration.

**What EV is about (2-3 sentence summary).** EV owns harmonik's event substrate: the typed event taxonomy (78 active types organized in eight thematic sub-tables), the common envelope, on-disk JSONL durability with class-driven fsync semantics (`fsync-boundary | ordinary | lossy-tail-ok`), the in-process pub/sub bus with the three-class consumer taxonomy (synchronous / asynchronous / observer), non-blocking back-pressure with per-class shed and overflow spill, observational-vs-state-reconstruction split, schema versioning with N-1 readability, monotonic UUIDv7 ordering with cross-restart high-water-mark, payload-redaction registry, and the Go tagged-union envelope + payload registry representation. EV is the canonical PAYLOAD-SHAPE owner for every cross-bus event the corpus emits; the WHEN of each emission is owned by the emitter spec per §6.5. EV depends on AR (axes / source_subsystem registration / amendment protocol), EM (Run/State/Transition scoping types and the state-reconstruction precedent), HC (ErrorCategory + redaction registry + handler-event emission rules — forward-deferred), and WM (session-log pipeline — forward-deferred).

## 4a. Event taxonomy task table (§8)

Per discipline §2.11(c), large taxonomies mint one bead per category under a taxonomy umbrella. EV's §8 has 78 active rows (+ 1 post-MVH-reserved row §8.6.14) across 8 thematic sub-tables. The umbrella `ev-events.taxonomy` is `parent-child` of the `ev` spec-parent; per-event rows are `parent-child` of the umbrella per §2.10's permitted second-level grouping.

Per-event bead pattern (mechanical):
- **bead-id:** `ev-events.<event-type-with-underscores-as-kebab>` (e.g., `ev-events.run-started`, `ev-events.run-failed`, `ev-events.checkpoint-written`).
- **title:** `Event row: <event_type> (§<x.y.z>)`
- **description:** "Per §<x.y.z>: implements `<event_type>` event-row registration. Durability class `<F | O | L>`. Emitter: `<emitter>`. Typical consumers: `<list>`. Payload fields: `<field-list-from-§8-row>`. Co-owned-WHEN: `<emitter-spec §<sec>>` per §6.5. The §8 row bead's work is (a) declaring the per-type payload struct in the Go registry per EV-032 (concrete YAML in §6.3 when listed; otherwise the §8 column field-list is normative); (b) registering the payload type at startup per EV-034; (c) ensuring the four-axis tag set + durability class per EV-031; (d) wiring the emitter call per the co-owned spec."
- **tags:** `mech, kind:event-row, req:EV-§<x.y.z>` plus axis tags per row's class (typically `axis:idempotency-non-idempotent` for F-class, `axis:idempotency-idempotent` for O/L most cases — full per-row axes registered in Go registry per EV-031).
- **blocks edges (canonical for every row):**
  - `ev-032` (registry tagged-union shape)
  - `ev-034` (startup-time registration)
  - `ev-031` (four-axis + durability tagging)
  - `ev-016` (durability class declared per row)
  - `ev-events.taxonomy` (umbrella — implicit via parent-child; F11 says no explicit blocks edge to umbrella for step beads, but §8 children are NOT step beads — they are §2.11(c) per-category beads under the umbrella per §2.10 permitted second-level grouping. F11 forbids explicit blocks-on-umbrella edges; the parent-child edge handles the relationship without explicit blocks).
  - PLUS row-specific edges per the row's `[execution-model.md ...]` / `[architecture.md ...]` / `[handler-contract.md ...]` cites (forward HC/WM cites for handler-event rows).

Per discipline §2.11(d) row-level dual-ownership annotations (where the §6.5 co-ownership map names a different OWNER for the WHEN), the row bead's description carries: "Co-owning emitter spec for this event's WHEN: `<spec>`; emission rule lives in `<spec>` §<sec>."
