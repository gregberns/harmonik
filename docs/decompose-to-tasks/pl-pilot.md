Read the discipline first; this file applies it.

**What PL is about (3-sentence summary).** PL defines harmonik's per-project headless daemon: each project (a git repo containing `.harmonik/`) runs exactly one daemon, identified by an advisory-locked pidfile at `.harmonik/daemon.pid` and serving JSON-RPC over a 0600-mode Unix socket at `.harmonik/daemon.sock`, with a deterministic 11-step startup sequence whose pre-classification orphan sweep (PL-006) clears residual tmux sessions / worktree locks / re-parented subprocesses bearing this project's provenance marker before any reconciliation detector fires. The daemon is a deterministic Go binary by structural rule — the composition root at `internal/daemon` instantiates every cross-subsystem registry (event bus, control-point registry, handler + skill registries, policy registry per PL-020a) and a `go-arch-lint` rule plus binary-level import-graph scan (PL-INV-002) forbids any LLM SDK from the daemon's transitive closure; cognition-bearing orchestrator-agents (post-MVH) sit on top via the CLI as separate Claude Code sessions (PL-019). Crash semantics route through stale-pidfile detection (`flock` failure plus `kill(pid, 0)` probe), startup re-runs reconciliation idempotently from step 0, the silent-hang detection obligation is named here but owned by HC §4.6, and `harmonik upgrade` performs an in-process `execve` with FD-passing (PL-027) under a `.harmonik/daemon.upgrading` marker (ON-020a-owned content) so client connections observe no `ECONNREFUSED` across the binary swap.
