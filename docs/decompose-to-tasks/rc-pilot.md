**What RC is about (one paragraph).** RC defines harmonik's reconciliation model — the restart-time and store-divergence-recovery contract that classifies every in-flight run into a bounded **11-category detection taxonomy** (Cat 0 infrastructure-unavailable; Cat 1 idempotent-rerun; Cat 2 non-idempotent-in-flight; Cat 3 store-disagreement-generic; Cat 3a torn-Beads-write; Cat 3b verdict-unexecuted; Cat 3c inverse-premature-close; Cat 4 recoverable-known-state; Cat 5 clean-restart; Cat 6a integrity-violation-LLM-triageable; Cat 6b integrity-violation-mechanically-unrecoverable), routes each category through an action-mapping dispatch contract (auto-resolver vs investigator-dispatch vs auto-escalate-to-operator), and runs reconciliation itself as an ordinary harmonik workflow (no separate subsystem) with bounded recursion guaranteed by the verdict-only-commit rule (RC-002). Detectors apply RC-003a's priority-ordered first-match resolution `Cat 0 → 6b → 6a → 5 → 3c → 3b → 3a → 3 → 2 → 4 → 1` to deterministically classify any in-flight run; investigator-required categories spawn an HC-handler agent bounded by a snapshot-token-captured logical view (RC-015) and a wall-clock budget (RC-017); the seven-value Verdict enum (RC-020) plus verdict-execution table is the deterministic dispatch surface for everything an investigator may emit; concurrent reconciliations against the same target run are serialized by an `flock(LOCK_EX|LOCK_NB)` advisory lock (RC-002a per PL-002a); evidence corroboration (RC-019a + RC-INV-004) ensures every `store_divergence_detected` carries a non-`inconclusive` corroboration value (single-source observations route to `divergence_inconclusive` instead). RC is foundation-cross-cutting: every other foundation spec is upstream of reconciliation, giving RC the highest cross-spec edge density in the corpus by design.
