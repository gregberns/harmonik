**What SH is about (3-sentence summary).** SH owns harmonik's end-to-end test rig: declarative YAML scenario files under `scenarios/` describing workflows + agent overrides + fixture setup + assertion vocabulary; the harness drives them through the same daemon entry-point and orchestrator startup as production via working-directory-assigned per-scenario synthetic project roots; twin-handler binaries (path-prefix discovered + HC-043 hash-verified) substitute for real-model handlers via handler-config override, never via runtime branches. Assertion evaluation is observational replay over the captured JSONL event log plus in-place workspace inspection; verdicts are typed (`pass | fail | timeout | error`) with an 8-class failure taxonomy (precedence-ordered). SH depends on AR (foundation amendment), HC (handler/twin contract surface, cancellation bounds, hash check), EV (observational replay, event taxonomy, bus_overflow escalation), WM (workspace primitive, lease model, squash-merge), EM (Outcome.status enum, failure-class analogs), ON (drain-timeout escalation), and PL (daemon lifecycle, socket RPC, one-daemon-per-project, orphan sweep markers).
