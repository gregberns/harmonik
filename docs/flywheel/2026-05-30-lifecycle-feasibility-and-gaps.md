# Flywheel Lifecycle — Feasibility & Gap Assessment (2026-05-30)

Synthesis of 5 parallel investigators after the 2026-05-30 credit-burn incident. Read-only forensics; no files were modified. All statuses are grounded in code/spec evidence verified against the live tree.

## Incident summary

On 2026-05-30 the user placed `ANTHROPIC_API_KEY` in a repo-root `.env` and within ~2 hours all API credit was consumed. The mechanism was not one runaway — it was **10 separate daemon boots** across the day (the last five within ~14 minutes: 14:42→14:50→14:52→14:54→14:56), each a bare `harmonik --project` daemon that **auto-pulled `br ready` into a fresh wave queue and dispatched immediately**, because the daemon has no idle / no-auto-pull mode and its default workloop falls back to a `br.Ready()` poll (`internal/daemon/workloop.go:833-858`). The leaked key reached every spawned claude via two unscrubbed env paths — the tmux substrate's additive `-e` flag over an inherited server env, and Pi's `syscall.Exec(resolved, command, os.Environ())` (`cmd/harmonik/supervise/shim.go:103`) — so claude billed pay-per-token API credits instead of the subscription. The one guardrail (`budget.ts` daily cap) defaulted to **Infinity** (`.pi/extensions/flywheel/index.ts:62-64`), tracked only Pi's own turns (not the daemon's claude children), and emitted zero budget events. Review-loop (on by default) roughly doubled sessions per bead (~23 implementers + 20 reviewers + 9 resumes in events alone). The daemon died around 14:56 leaving durable wreckage: an orphaned active wave queue (`019e7ae3...`, one bead `failed`), 6 beads stuck `in_progress`, 11 orphaned run-worktrees, 24 stale `*-flywheel` tmux sessions, and **8 claude sessions still running and still spending** at forensics time.

## Recovered lost issues (no open bead before this assessment)

Already correctly captured (do **not** re-file): hk-rwsg9 (cache-warmth), hk-exd7m (daemon idle/no-auto-pull, in_progress), hk-vh1jc (reviewer false-approve), hk-3w50w (daemon-revival), hk-m8zqv (smoke), hk-dgwf4 (P0 exit-17 crash), hk-hlmup (P1 exit-17), hk-iuaed (imrest orphan-sweep spec, post-mvh).

Genuinely lost — surfaced by the incident, never converted to a bead:

1. **No spend ceiling on the GO DAEMON's claude implementer/reviewer sessions** (the literal root cause). `budget.ts`/CL-090 governs only Pi's own model turns; the 26+ implementer sessions that burned the credit are invisible to it. No per-day-USD / max-runs cap exists daemon-side.
2. **`ANTHROPIC_API_KEY` reaches every spawned claude unscrubbed** — via the tmux substrate's additive `-e` over an inherited server env (`osadapter.go:497`) and via Pi's `os.Environ()` passthrough (`shim.go:103`). No code anywhere scrubs the key from child env.
3. **`.env` holding the key is not gitignored** (`git check-ignore .env` → exit 1; no `.env` in `.gitignore`). A stray `git add -A` would commit the secret.
4. **`supervise start` does not inject the API key** into the Pi — silent auth failure on every fresh boot (HANDOFF blocker #1, never filed).
5. **The orphan sweep is non-remediating** — the 14:56 sweep observed 951 stale intents + 6 stuck in_progress + 24 tmux sessions and reset/killed **zero** (the imrest reset is post-mvh-deferred). Plus it never reaps orphaned `*-flywheel` tmux sessions or worktrees.
6. **Pi judgment model hardcoded to opus + no runtime override** (`router.ts:84`, `claude-opus-4-7-…` literal); no `FLYWHEEL_MODEL_TIER*` env knob symmetric with the budget knob. The daemon's claude baseline (sonnet/medium) is likewise only changeable by editing Go or per-project config with no hot-reload.
7. **No global re-dispatch/retry budget** on the review-loop / `no_progress_detected` path — each retry is a full paid claude session; the queue only paused at fail_count:4. (hk-63oh.38 caps only Cat-3b reconciliation.)
8. **Dual-orchestrator collision** — killing the auto-pull daemon tripped a *separate* live `harmonik-pi` Pi doing the same work. No single-flywheel-per-project lock.
9. **Crash-leaked live processes** — 8 `claude --session-id 019e7a*` sessions + a `pi` process were still running and spending at forensics time; dead daemons leave no reaper.
10. **No `--dry-run`/plan-only daemon mode** to preview "this will spawn N implementers + N reviewers across M beads" before touching a live key.

## Recommended kerf works

The "make the desired lifecycle actually possible" effort is too large and cross-cutting for a single bead. It splits cleanly into three works because they touch three different subsystems with different review surfaces:

1. **`credfence`** (jig: spec) — Credential & spend safety fence. Defines the normative contract for credential isolation (which process may hold the key, scrub guarantees at every spawn boundary) and a unified spend-governance model that covers BOTH Pi turns AND daemon-spawned claude sessions (per-day USD, max-runs, finite default, hard-halt wiring into the existing `handlerpause` `budget_exhausted` consumer). Prime incident-prevention work; the P0 beads above are its first tasks. Scope: `internal/daemon/tmuxsubstrate.go`, `cmd/harmonik/supervise/shim.go`, `.pi/extensions/flywheel/budget.ts`, `specs/cognition-loop.md`.

2. **`pilot`** (jig: spec) — Pi-driven dispatch & control plane. Closes S2/S5/S6/S9/S10 as one coherent contract: a quiet-by-default daemon (no-auto-pull), a Pi-side curated-dispatch harness that calls `queue submit`/`append` (implementing CL-070..073), and an agent-callable pause/resume control surface with a real `operator_pause_status` producer. This is where "the lifecycle the user actually wants" is specified end-to-end and the submit/append/pause CLI verbs are designed. Scope: `internal/daemon/workloop.go`, `cmd/harmonik/supervise_cmd.go`, `.pi/extensions/flywheel/{bridge,router}.ts`, `specs/{execution-model,cognition-loop,queue-model}.md`.

3. **`reap`** (jig: bug) — Crash-recovery & orphan remediation. Makes the orphan sweep actually remediating (reset stale `in_progress`, kill orphaned `*-flywheel` tmux sessions + worktrees, reconcile `dispatched` items on boot), adds reap-on-exit, restart backoff, and the single-flywheel-per-project lock. Lifts the post-mvh imrest deferral (hk-iuaed) into a real implementation. Scope: the daemon orphan-sweep path + `internal/supervise/`.

These three are sequenceable but largely parallel: `credfence` is the safety prerequisite (do first), `pilot` is the lifecycle build-out, `reap` hardens the failure modes.
