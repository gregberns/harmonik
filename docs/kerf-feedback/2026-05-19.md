# Kerf Feedback — 2026-05-19

## KERF-UPSTREAM: Jig templates must require a Validation / Acceptance Tests section

**Tag:** MAJOR  
**Component:** All built-in jigs (plan, spec, bug, implementation)  
**Filed by:** hk-85trr (agent-dispatch, 2026-05-19)

### Requirement

Every kerf jig that produces a normative planning artifact (change spec, spec draft, fix spec, task list, or implementation breakdown) MUST require a **"Validation / Acceptance Tests"** section at the end of that pass. The section must mandate:

1. At least one **scenario-test bead** — covering the end-to-end workflow using a twin or real-claude substrate. The bead must name the CLI invocation, the bead lifecycle state being exercised, and the observable terminal condition.
2. At least one **exploratory-test bead** — covering the operator-facing surface (CLI command, API call, or output a human or submitter agent would actually invoke).

Both beads must be listed by ID in the artifact and must remain open until explicitly closed. Neither the plan/spec/bug work nor its downstream implementation beads may close before the test beads do.

### Motivation: the half-built-systems pattern

Three incidents from the 2026-05-18 harmonik dogfood session illustrate the pattern:

- **hk-37zy8** (handler-pause policy goroutine): the goroutine was unit-tested, the bead was reviewer-APPROVED, and it was committed — but the goroutine was never wired into the composition root. A twin-based scenario test verifying that `harmonik run` dispatching a `handler_fatal` bead actually emits `handler_paused` and writes `.harmonik/handler-state.json` would have caught the gap at PR time. Instead it required a separate fixup bead (hk-c8k4c).
- **hk-aievp** (DOGFOOD-BLOCKER: daemon paste-injects task to STALE pane — prior-session `lastHandle` pollution): discovered during live dogfood run. No scenario test existed for the pane-reuse path.
- **hk-ry3be** (DOGFOOD-BLOCKER: daemon emits heartbeats for ~15 hours after claude pane disappears — no agent-termination detection): discovered during live dogfood run. No scenario test existed for the agent-disappearance path.

The common factor: implementation was correct at the unit level, reviewer approved the code, but the integration surface was never exercised. Kerf jigs currently produce no structural requirement for this. Adding a mandatory Validation section closes the gap at the planning stage.

### What was done locally (harmonik-side, pending upstream adoption)

Local jig overrides have been installed at `.kerf/jigs/` (tracked in the harmonik repo) for all four built-in jigs:

| Jig | Pass(es) where Validation section was added |
|-----|---------------------------------------------|
| `plan` | Pass 5 (Change Spec) and Pass 7 (Tasks) |
| `spec` | Pass 5 (Spec Draft) and Pass 7 (Tasks) |
| `bug` | Pass 5 (Fix Spec) |
| `implementation` | Pass 1 (Breakdown) and Pass 4 (Verify) |

Each Validation section cites hk-37zy8, hk-aievp, and hk-ry3be as motivating examples and specifies the bead title conventions (`scenario: <codename> — <brief>` and `explore: <codename> — <brief>`), the labels to use (`--label scenario-test`, `--label exploratory-test`), and the dependency-graph requirement (test beads must be dependents of the implementation beads they validate).

### Suggested upstream change

Add a `validation` field to each pass in the jig YAML schema, or add a new pass type `validation` with structural enforcement. At minimum, add the Validation section to the agent instructions for each relevant pass in the built-in jig text.

The plans/README.md "Done means..." convention in harmonik already encodes this requirement for plans:

> Each plan MUST list at least one scenario-test bead covering the end-to-end workflow (using a twin or real-claude substrate), and at least one exploratory-test bead for the operator-facing surface...

The jig agent instructions should be the canonical enforcement point, not just the plan template.

## BEADS-RUST-UPSTREAM: br doesn't auto-checkpoint WAL; writes degrade from 0.35s to 19s as WAL grows

**Tag:** MAJOR
**Component:** beads_rust (`br` CLI), SQLite WAL management
**Filed by:** hk-5dewt (agent-dispatch, 2026-05-19)

### Symptom

After 4 harmonik dogfood runs, every `br close` in the daemon's bead-lifecycle path
timed out. Manual diagnosis:

- `br close <id>` wall-clock time: **19.4 seconds** with a 32 MB DB + 12 MB un-checkpointed WAL.
- `br close <id>` wall-clock time: **0.35 seconds** immediately after `sqlite3 .beads/beads.db "PRAGMA wal_checkpoint(TRUNCATE);"`.
- Read commands (`br list`, `br show`) remained fast (0.16–0.18 s) throughout — only writes were affected.

### Root cause

SQLite's WAL mode allows reads to continue while the WAL file accumulates write
pages. Each write command must scan the entire WAL to determine which frames are
current before committing, so write latency grows O(WAL size). `br` does not issue
`PRAGMA wal_checkpoint` after its own writes, so the WAL grows unbounded between
restarts. At 12 MB the per-write scan cost reached ~19 s, exceeding the 10 s
wall-clock timeout in harmonik's brcli adapter.

### Workaround applied in harmonik (pending upstream fix)

`internal/daemon/walcheckpoint.go` (bead hk-5dewt) runs
`PRAGMA wal_checkpoint(TRUNCATE)` via the `sqlite3` CLI at daemon startup when
`.beads/beads.db-wal` exceeds 1 MB. This is a host-side workaround — it requires
`sqlite3` on PATH and fires once per daemon start rather than incrementally.

### Suggested upstream fix in beads_rust

Option A (preferred): after every terminal-transition write (`br close`, `br update
--status closed`) issue `PRAGMA wal_checkpoint(PASSIVE)` via the Rusqlite
connection. PASSIVE mode is non-blocking and will checkpoint as much as possible
without interrupting concurrent readers.

Option B: checkpoint at `br` startup (before the first write) when the WAL exceeds
a configurable threshold (e.g. 1 MB). This mirrors the harmonik workaround but
is contained within `br` itself.

Option C: enable SQLite's auto-checkpoint pragma (`PRAGMA wal_autocheckpoint=1000`)
which triggers a PASSIVE checkpoint automatically after every 1000 WAL pages
(~4 MB). This is already the SQLite default when using the C library but may be
disabled by Rusqlite configuration.

Any of the three options would restore sub-second write latency as the corpus
grows past 32 MB.

### Impact

P0 for harmonik: every dogfood run since 2026-05-15 has required a manual
`br close` after each bead dispatch because the daemon's 10 s timeout fires
before the WAL-bloated write completes. The workaround in hk-5dewt mitigates this
but does not eliminate the root cause — WAL will re-accumulate between daemon
restarts if `br` continues to write without checkpointing.

## CORRECTION: WAL was NOT the root cause — .br_history bloat is

**Filed by:** hk-5dewt (follow-up, 2026-05-19)

### Correction

The WAL-checkpoint pre-flight added at commit abf36c4 (hk-5dewt wave 1) did NOT
fix the 19.5 s `br close` latency.  Post-fix dogfood run #5b showed the identical
failure pattern: 11/11 BrUnavailable timeouts.  Manual re-test with WAL at 604 KB
(well below the 12 MB pre-fix level) still clocked 19.5 s.  WAL size is not the
cause.

### Actual root cause: .br_history/ snapshot bloat

Each `br` write appends a **~1.2 MB snapshot** to `.beads/.br_history/`.
With 200+ entries (226 MB total), br must enumerate and scan the entire directory on
every write.  The per-write scan cost at 200 entries is ~19.5 s.

Validation: archiving all but the 5 most-recent entries reduced `br close` to
**0.15 s** — a 130× improvement.

### Fix applied in harmonik (hk-5dewt wave 2)

`internal/daemon/brhistoryrotate.go` implements `runBrHistoryRotationPreflight`:
- At daemon startup, immediately after the WAL-checkpoint pre-flight, stat
  `.beads/.br_history/` entry count.
- If entry count > 20 (keepLatest), rename the oldest entries into a sibling
  `.beads/.br_history-archive/` directory with `.archived-<ts>` suffixes.
- Logs structured events: `br_history_rotation_started total=N keep=K`,
  `br_history_rotation_completed archived=M remaining=K duration_ms=Y`.
- Non-fatal: errors log warnings and return nil.
- `Config.SkipBrHistoryRotation bool` added for test isolation.

The WAL pre-flight at abf36c4 remains as defense-in-depth (WAL will still grow
between daemon restarts; the checkpoint resets it each start), but it is NOT what
was causing the 19.5 s latency.

### Suggested upstream change in beads_rust

`br` should auto-rotate `.br_history/` after reaching a configurable entry cap
(e.g. 50 entries), or provide a `br history prune` subcommand, so that harmonik
does not need to manage the directory itself.  Per-write ~1.2 MB snapshots will
eventually re-accumulate at the 20-entry cap (~24 MB) between daemon restarts if
harmonik runs many writes; a lower entry cap or a size-based rotation policy would
keep write latency permanently bounded.
