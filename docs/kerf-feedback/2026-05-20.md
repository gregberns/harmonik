The triage suggester correctly proposes `kerf pin bridge-integration <id>` for some but that's not the right home; they're a separate stream.
   **Suggested fix on the harmonik side:** create a work `kerf-upstream` with `bead-filter 'label=kerf-upstream'` to surface them as a coherent stream in `kerf next`. Will do this next session unless feedback says otherwise.

This is hk-u5ymq (already filed). Not a new issue; just re-confirming the pattern is common in the v51 corpus.

Suggestion to upstream: a `kerf feedback list --since <baseline>` command that emits the rollup. Until then, `br list --label kerf-upstream --status open` works.

## OBSERVATION: `kerf next` does NOT show priority

The ranked feed has 49 dispatchable beads but no priority column. The orchestrator has to cross-reference `br list --priority N` to know which top-ranked items are P1 vs P2 vs P3. Suggestion: add a `[P1]` / `[P2]` annotation in the text output, or a `--priority-filter` flag. This matters specifically for the "harmonik daily loop" — the agent should not have to run a second query to know what to dispatch.
