## Issue 1 — `kerf square` filename expectations don't match component-prefixed drafts

**Severity:** medium (blocks `kerf square` until manually worked around).

**Symptom.** `kerf square phase-3-dot` reported `Files: fail — 11/22 expected files present` with the missing list:

    SESSION.md
    04-design/execution-model-dot-design.md
    04-design/handler-contract-outcome-design.md
    04-design/workflow-graph-design.md
    04-design/control-points-binding-design.md
    04-design/examples-design.md
    05-spec-drafts/execution-model-dot.md
    05-spec-drafts/handler-contract-outcome.md
    05-spec-drafts/workflow-graph.md
    05-spec-drafts/control-points-binding.md
    05-spec-drafts/examples.md

All five spec drafts and four of five design docs existed under `C1-`/`C2-`/`C3-`/`C5-` prefixed names. Kerf's square check expected component-name-only paths.

**Workaround.** Symlinked each `Cn-<name>.md` to `<name>.md`. Wrote a stub `SESSION.md` to satisfy the file-presence check.

**Why this happened.** Pass-2 (decompose) and pass-3 (research) explicitly used `C1..C5` shorthand to talk about components. Pass-4 and pass-5 reviewers reinforced the prefix in artifact filenames so the changelog could trace "C1 → workflow-graph", "C2 → execution-model-dot", etc. That's a reasonable convention — but `kerf square` doesn't know about it.

**Suggested fix (one of).**
- (a) `kerf square` reads the component-name list from `02-components.md` and accepts any file matching `<anything>-<component>.md` in the expected directory.
- (b) `kerf square` accepts a `<workdir>/aliases.yaml` mapping expected name → actual file.
- (c) Document the canonical "name files exactly `<component>.md`" convention in the jig output at pass-4/5, so authors don't invent a prefix scheme.

(a) is the least invasive — pass-4/5 reviewers can keep enforcing the prefix discipline for narrative clarity while square accepts the actual files.
