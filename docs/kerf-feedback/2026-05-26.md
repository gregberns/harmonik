# Kerf Feedback — 2026-05-26

## KF-026-01: `kerf next` inter-work ranking doesn't reflect operator priority

**Observed:** DOT beads (label `codename:phase-3-dot`, all P1) appear at positions 29-55 in `kerf next`, behind `phase-2-completion` (positions 1-20) and `handler-pause` (21-28).

**Root cause investigation:** `kerf next` appears to group/sort by work, not by individual bead priority. All three works' beads are P1, so bead-level priority doesn't differentiate. The ordering is driven by kerf's internal work-ranking heuristic (likely creation date or status progression). `kerf map` shows all three works are in "unassigned" area, so area-based weighting isn't a factor either. There is no `priority` field on kerf works, no `--work` filter on `kerf next`, and no way for the operator to express "this session I care about work X." The ranking is opaque and not configurable.

**Impact:** Operator must bypass `kerf next` and dispatch DOT beads by ID, which defeats kerf's role as the priority source of truth (per HANDOFF.md hard rule).

**Expected:** Operator should be able to boost a work's priority in the feed, or scope `kerf next` to a single work.

**Workaround:** Dispatch DOT beads explicitly via `harmonik run --beads` bypassing `kerf next` ranking.

**Upstream fix candidates:**
1. `kerf work edit <name>` gains a `priority` field that influences `kerf next` ranking.
2. `kerf next --work <name>` filters the feed to a single work's beads.
3. `kerf next --label <label>` filters by bead label.
4. `kerf next` exposes its ranking heuristic so operators can understand and tune it.

**Label:** `kerf-upstream`

## KF-026-02: `kerf next` scoring ignores `br` priority entirely (hk-f7fnt)

**Observed:** In v53, 50+ beads were tied at identical score (3.486). P0 and P3 beads are indistinguishable in the feed. In v60 cross-session analysis, `kerf next` returns 75 beads but neither systemic throughput blocker (empty-pane, close-without-impl) appears — because those had no beads filed. Even after beads ARE filed at P0/P1, they won't surface at the top unless kerf factors `br` priority into scoring.

**Root cause:** `kerf next` ranking heuristic does not read or weight the `priority` field from `.beads/beads.jsonl`. Scoring appears to use only graph structure (dependencies, work membership, staleness).

**Impact:** The orchestrator's HANDOFF says "KERF IS THE PRIORITY SOURCE OF TRUTH" but in practice the orchestrator dispatches by explicit bead ID because kerf's ordering doesn't reflect urgency. This defeats kerf's intended role.

**Expected:** `kerf next` should factor bead priority into its ranking score. P0 beads should always appear before P3 beads, regardless of graph position.

**Upstream fix candidates:**
1. Multiply kerf's internal score by a priority weight (e.g., P0=4x, P1=3x, P2=2x, P3=1x, P4=0.5x)
2. Add `--min-priority` / `--max-priority` filters to `kerf next`
3. Surface priority in `kerf next` output so the operator can at least see it

**Label:** `kerf-upstream`
