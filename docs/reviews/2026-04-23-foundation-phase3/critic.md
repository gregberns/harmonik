# Critic Review — Foundation 2026-04-21

## Summary verdict

The 10-component decomposition reflects the 2026-04-20/21 decisions with real care — the three-store rule, the reconciliation-as-workflow principle, and the handler/daemon split are load-bearing and now named. But the document is a decompose artifact doing spec-draft work in several places, and much of the newly absorbed content smuggles normative claims in on verbs that don't specify their mechanics: "classifies," "reconciles," "propagates," "wins." The overall arc is sound; the weakest surfaces are exactly the places the 2026-04-20/21 decisions bear weight — reconciliation detection, the three-store arbitration procedure, Beads-CLI-as-boundary, and the startup sequence. Several claims are provably under-specified because they reference peer specs that also under-specify them (circular "owned by the other spec" gestures). The review below is blocker-heavy; that's appropriate for a component doc that now carries architectural decisions rather than just a file list.

## Critical findings (under-specification that blocks spec-draft)

- **§9.3 Cat 3 detectors — procedure absent.** "Multiple pattern-specific sub-detectors (premature-close, manual-close, missing-worktree, duplicate transition_id)" is the entire normative surface for the most common store-divergence class. The doc does not specify: (a) the order detectors run (does premature-close test before missing-worktree, or in parallel?); (b) what happens when two sub-detectors both fire on the same bead (Cat 3 + Cat 6, or Cat 3a + Cat 3b); (c) whether the detector queries Beads at read-time or from a snapshot taken at startup (matters if beads mutate during the walk); (d) the exact Beads query shape (`br list --status in_progress`? `br ready`? both?). Spec-draft cannot write this section because the detector function's input/output and ordering aren't resolved.

- **§8.2 startup sequence — step 4 "builds an in-memory model" is hand-waved.** The step says the daemon "builds an in-memory model of: completions (beads with corresponding merge-to-main commits), open beads (pending), in-flight beads." Nothing specifies how "corresponding merge-to-main commits" are detected. Are we grepping git log for `Harmonik-Bead-ID` trailers? Walking branches and matching via branch-name convention (§5.8)? What about squash-merged commits whose trailer was dropped? What about main commits that don't carry the trailer because they pre-date harmonik? The algorithm is load-bearing for correctness (a miss classifies a completed bead as in-flight, triggering a spurious reconciliation workflow).

- **§2.6 "git wins on completion" — arbitration procedure undefined.** The rule is stated three times (§2.6, §9.2 Cat 3, §10.7) but the *procedure* isn't. If Beads says `closed` and git has no merge commit, does the daemon (a) reopen the bead automatically, (b) spawn an investigator, (c) both? §10.7 says "Beads status is corrected after the investigator's verdict, NOT silently." §9.2 Cat 3 says "Rule: **git wins on completion facts** … investigator workflow explains the divergence and corrects the cache." These are consistent if you squint, but: which spec owns the write? §10.4 says harmonik writes to Beads only on claim/close/reopen — is "correct Beads status after investigator verdict" a fourth write kind, or is it one of the three? What verdict from §9.5 covers this (probably `reopen-bead`, but the doc doesn't say)?

- **§9.2 Cat 1 — "idempotent" is assumed, not demonstrated.** "Reviewer agent mid-run, researcher agent" — how does the detector KNOW the node was idempotent? The node_id has to be tagged. Where does the tag live? The doc implies a "node type tagged as idempotent" (§9.3 Cat 1 detector language), but execution-model §2.1 defines node `type` as the enum `{agentic, non-agentic, gate, control-point, sub-workflow}`. None of those tell you idempotency. The four-axis tags (architecture §1.1) include an idempotency axis, but those tags are on types/interfaces/evaluation points, not on workflow nodes. The detector needs a node-level idempotency tag that doesn't exist in execution-model §2.1.

- **§8.2 step 2 — "walks the git log for the project's repo." Scope is unbounded.** Every startup walks every commit that ever carried a `Harmonik-Run-ID` trailer? That's the entire project history. The doc gestures at §7.8 Criterion 2 ("git-log walk depth since the oldest open-bead's first checkpoint") but process-lifecycle.md doesn't reference that optimization — it just says "walks the git log." Either the walk is bounded by the oldest open bead (in which case process-lifecycle needs to specify how that bound is computed), or it's unbounded (in which case §7.8's 30-second RTO is aspirational on any mature repo).

- **§9.4 "Agent session log if one exists (CASS-indexed per workspace-model.md §5.3)."** The investigator is declared to read CASS-indexed content, but §5.3 says CASS indexing "follows these directories via a configured set of roots" — there's no read API specified, no query shape, no freshness guarantee. If the daemon just crashed, is the most recent session log indexed yet? What happens when the investigator needs content that hasn't been indexed? The spec doesn't say the investigator reads the raw session-log files (which would be simpler and not depend on CASS freshness).

- **§7.3 "Reconciliation carve-out" creates an indeterminate state.** "An operator pause issued during `reconciling` is queued and applied when the reconciliation batch completes (boundary event: all reconciliation runs either resume into normal flow or produce a verdict)." What if reconciliation workflows themselves take 10 minutes because they're waiting on investigator LLM calls? The operator issued `pause`; they expect the system to pause. Is there an operator-observable signal "your pause is queued behind N reconciliation runs, longest-running is X seconds"? §8.2 promises the daemon reaches `ready` before normal workflow dispatch begins, but §9.1 says reconciliation workflows run under the daemon. Is the daemon `ready` *before* reconciliation workflows complete, or *after*? If before, the between-task invariant (§7.3) applies — but then why is there a carve-out? The state machine is internally inconsistent.

- **§4.11 "fail-launch if a required skill cannot be provisioned" — ambiguous routing.** The handler emits `ErrSkillProvisioningFailed`, which is a category not in §4.5's typed-error list (`ErrTransient`, `ErrStructural`, `ErrDeterministic`, `ErrCanceled`, `ErrBudget`). Does it map to `ErrDeterministic`? `ErrStructural`? The failure-taxonomy classifier (§2.3, mechanism-tagged) needs a deterministic answer; "handler emits a new typed error" doesn't tell the classifier anything.

- **§10.4 "Close: in_progress → closed" triggering is under-specified.** "Occurs when the run's workflow reaches a success terminal state AND the merge to the target branch has completed (per workspace-model.md §5.8 branching model)." What's a "success terminal state"? A node with no outgoing edges? A node marked terminal in DOT? What if the workflow graph has multiple terminal nodes (some success, some failure)? How is "the merge has completed" detected — by the `workspace_merged` event, by polling git, by a return value? And is the close a daemon write or an agent-triggered write (since §10.2 says the daemon "invokes `br` directly … for its terminal-transition writes")?

- **§2.1 `transition` and §2.1 `checkpoint` — storage relationship is mushy.** §2.1 says the checkpoint commit carries `Harmonik-Run-ID`, `Harmonik-State-ID`, `Harmonik-Transition-ID` trailers. §2.2 says "the canonical durable record is the `transition` … stored as part of the checkpoint-trail (a git commit, per §2.1) with the full AlphaGo fields (prior state, actor role, candidate actions, chosen action, policy version, evidence, verifier metrics, next state, confidence)." Where? In the commit message body? As a file in the tree? As a git note? The trailer format is key-value; it can't carry `candidate_actions[]` as a structured list. Spec-draft can't proceed without knowing the physical storage of the trace record.

## Important findings (hand-waving / assumptions / missing edge cases)

- **Implicit: git commits are immutable for the life of a run.** The three-store rule and the reconciliation story both assume no history rewrite of the project repo. Nobody `git reset --hard`s a branch that has in-flight checkpoints; nobody force-pushes over a task branch. This should be stated as a harmonik precondition — the daemon operates on the assumption that *its* commits are never rewritten. What happens if they are? (Reconciliation would see a commit disappear, trigger Cat 6, escalate to human.) But the assumption itself isn't named.

- **Implicit: Beads bead IDs are stable forever.** §10.3 says "stable for the lifetime of the bead," but harmonik's artifacts embed bead IDs in git trailers (§2.1), event payloads (§3.2), and session-log metadata (§5.3). If a bead is tombstoned and its ID is reused (does Beads reuse? — external component, external rule), harmonik's historical references silently repoint. The spec should state a precondition: bead IDs are globally unique *and never reused*, even post-tombstone.

- **Implicit: a single daemon per project is enforced.** §8.2 step 1 says "if another daemon is already running for this project, the new process exits with a specific error code." But what enforces this on a stale pidfile? §8.8 says a stale pidfile is detected "by checking the PID is no longer a live process" — classic TOCTOU race: PID N was the old daemon, N terminates, `harmonik daemon` starts, PID N is reassigned to an unrelated process, the daemon thinks the old daemon is still alive and exits. `flock`? `fcntl` advisory lock? The mechanism is unnamed.

- **Implicit: workspaces survive daemon restart.** §9.2 Cat 6 names "unexpected workspace loss (workspace path no longer exists on disk)" as a detector signal, which implies the daemon assumes workspaces DO survive restarts. But nothing says the worktrees are on stable storage vs a tmpfs. Nothing specifies cleanup responsibility on unexpected restart. An operator who `rm -rf`s a worktree out-of-band triggers Cat 6 for every bead that was leased there.

- **§9.8 "Crash-recovery-tested" cites `docs/methodology/TESTING.md` as the binding test layer.** Is TESTING.md a foundation-level spec now? It wasn't in the dependency graph. If reconciliation depends on a testing methodology doc for correctness, that dependency should be explicit. (Alternatively, the testing methodology reference is aspirational — in which case §9.8 isn't actually enforceable.)

- **§7.8 "Reconciliation-workflow dispatch time is separate from the RTO target."** Reasonable in principle, but observable behavior for the operator is confused. If I restart and 30 seconds later the daemon is `ready` but 12 investigator workflows are running in the background, can I enqueue new work? §8.9 says the between-task invariant applies "only once the daemon reaches `ready`" — OK, so enqueues go. But §7.3's reconciliation carve-out says operator pause during `reconciling` is queued. These are two different states the operator interacts with differently, both after `ready`. The operator's mental model is unspecified.

- **§2.2 transition-as-projection — what if the projection diverges?** "A `transition_event` is emitted to the event bus as a projection of the transition record. The event payload cites the transition by `transition_id`; the full record is recoverable by resolving the checkpoint reference." What if the projection is written but the checkpoint commit fails to create (disk full, repo locked, etc.)? Then we have a transition_event in JSONL whose checkpoint reference resolves to nothing. Per §3.6 JSONL isn't state-authoritative, so this is "OK" in some sense — but what Cat does it become at next restart? Cat 6? §9.2 Cat 6 says "a checkpoint commit referenced in JSONL is missing from git" — yes, this case. But the event was produced BY harmonik; it's a harmonik bug, not a corruption event. The spec should distinguish external corruption from internal protocol violation.

- **§6.6 "Default skills" — skill versioning for the Beads-CLI skill unaddressed.** The Beads-CLI skill is "cited" at §10.9 without being defined. But harmonik pins Beads version (§10.8). A Beads version bump means the skill content changes. Is the skill versioned? Tied to a harmonik release? If the operator pulls a newer skill into an older harmonik, does the handler-contract.md §4.10 commit-hash check cover the skill path? The skill injection is load-bearing (it's the only Beads access path for agents) and its integrity model is missing.

- **§7.3 "pause → paused" drain semantics for long-running agents.** "Between-task invariant: pause and upgrade complete in-flight runs before taking effect." Real agents sit in budget-bounded LLM calls that can take minutes. If I pause at minute 10 of a 15-minute LLM call, I wait 5 minutes? 15 minutes if the agent retries? The upgrade pathway depends on this. §7.7 graceful shutdown mentions drain timeout but doesn't specify the default. If it's e.g. 60s, then pause cannot wait for a 15-min LLM call — pause-to-upgrade fails graceful and becomes immediate (which the spec doesn't permit; immediate is a separate mode). Interaction between drain timeout and between-task invariant needs resolution.

- **§10.5 "Ready-work queries — `br ready` (or equivalent)."** "Or equivalent" is hand-waving in a normative contract. Either it's `br ready` or it's a specific sequence of commands; "or equivalent" means no reviewer can grep for compliance. SC-8 bans "as needed" language; "or equivalent" is in the same family.

- **§3.4 "fsync is called at every run-boundary and at every checkpoint_written event."** OK — but §3.7's dead-letter queue is a separate JSONL file (`dead-letters.jsonl`). Is fsync of dead-letters tied to the same cadence? If a dead-letter is written but not fsync'd, and the daemon crashes, the async-consumer failure is lost silently. §3.7 says "persistent JSONL file … per the same durability contract as the main event log (§3.4)," but §3.4 doesn't say anything about dead-letter specifically. Operator of a dead-letter replay tool needs to know what happened between the last fsync and the crash.

## Nits

- **§2.1 edge cascade order: "condition match → label match → preferred_label → weight → lexical order."** `preferred_label` is a field on `edge` AND on `outcome` (routing hint). Is "label match" testing against the outcome's `preferred_label`, or the edge's `preferred_label`? The cascade says both "label match" and "preferred_label" as separate tiers, implying different things, but only one `preferred_label` field exists in the spec. Naming clash with itself.

- **§3.2 event taxonomy mixes granularity casually.** `operator_pausing`, `operator_paused`, `operator_resuming`, `operator_stopped`, `operator_upgrading` — five events for five state transitions. But `run_started`, `run_completed`, `run_failed` — three for a state machine that also has paused/quarantined states elsewhere. Inconsistent granularity.

- **§8.3 `harmonik runner` — "optionally spawns an orchestrator-agent session."** The orchestrator-agent is architecturally distinct from the daemon (§8.6) and MUST NOT share process space. But `runner` spawning it as a convenience means solo-dev ergonomics muddies the distinction in practice. Worth an explicit note: the orchestrator-agent launched by `runner` is still a separate process.

- **§1.9 "a bead references the workflow + node it is executing."** Beads beads reference workflow+node? That's harmonik overlay data. Per §10.4, harmonik does not write intra-run state to Beads. Where does this reference live? In the bead's description field (freeform text)? In a harmonik-side map? In a Beads custom field? Cross-reference to §10 leaves this unspecified.

- **§4.2 "spec size. If MVH LaunchSpec size stays under 1 MiB (expected), Option A. Otherwise Option B. Research pass measures."** Good criterion, good deferral. But "Option A … preferred for short-lived handlers" and "Option B … preferred for long-lived handlers" inside the same bullet also mixes spec-size and lifetime as decision criteria. Which dominates? (Presumably size, but the text doesn't say.)

- **§7.8 "Criterion 3 — hard ceiling. 300 seconds."** Hard-ceiling-ish. A "degraded state that reports `reconciling` with progress markers" is still reconciling, still not `ready`. Is the 300s ceiling *the RTO failed* or *the operator is paged*? The spec is quiet on operator consequence at boundary violation.

- **§1.8 "Merge responsibility: a merge agent (when the workflow uses a distinct merge node)."** "When" is load-bearing. Some workflows have no distinct merge node (the implementer commits directly, no review). Foundation should say: if no merge node, who performs the merge-to-integration? §10.4 close trigger says "merge to the target branch has completed" — by whom, if not a merge agent? Probably the implementer, but not named.

## Affirmations

- **The three-artifact separation (§1.9) and the naming-conflict resolution for "bead" vs "run" (§2.5) are strong.** The "feature is not a primitive" position is declared and defended; the document doesn't let the concept sneak back in at §8 or §10.

- **The observational-vs-state-reconstruction replay split (§3.6) is rigorous.** "JSONL is never walked to reconstruct in-memory workflow state" is the single hardest thing the no-DTW decision requires foundation to say out loud, and the doc says it cleanly with appropriate consumer-side obligations (idempotency contract for observational replay).

- **The handler-as-modularity-boundary framing (§4.12) is excellent.** Explicit naming that ntm is on one side of the boundary and the daemon on the other — with the prediction that the boundary must survive execution-shape evolution — is exactly the kind of load-bearing commitment that protects the design from the first wave of implementation pressure.

- **The control-points primitive table (§6.1a) successfully unifies three notions without collapsing them.** The per-Kind columns (trigger, evaluator-input, evaluator-returns, outcome-action, boundary-classification rule) give reviewers something to grep and reviewers in other specs something to cite. The mechanism-only rule for Guards is sharp and defensible.

- **Reconciliation-as-workflow (§9.1) is a structurally elegant answer to the no-DTW question.** It reuses the primitives, which means "testing reconciliation" reduces to "test workflows" plus the category detector. That said — §9.3's detector is where the elegance has to land, and that's where the top-of-list critical finding lives.

- **The store-authority rule (§2.6 + §10.7) is *stated* clearly** — "git wins on completion disagreement." The problem is the arbitration procedure, not the rule. The rule itself is correct and defensible.
