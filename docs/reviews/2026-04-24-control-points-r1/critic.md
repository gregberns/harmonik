# Round 1 Critic Review — control-points.md v0.1

## Verdict summary

The draft is structurally clean and does the right work unifying Gate / Hook / Guard registration and ownership (S02 registry, S01 invocation, S05 dispatch). The load-bearing softness sits in four places: the promotion of Budget from a §6.9-style enforcement point in components.md to a fourth Kind peer with Gate is underjustified (the §4.1.CP-005 table shows the Kinds are NOT actually peers — Budget's trigger, evaluator input, and enforcement locus share essentially nothing with the other three); the adoption of `expr-lang/expr` names the right criteria in rationale but never engages the one serious alternative (cel-go) that meets the same criteria; the persisted-verdict replay-safety story treats "model call avoided" as equivalent to "replay is safe" when the verdict's INPUT envelope (context, skills, prompt template version) is what actually needs pinning; and the "default permission sets" for MVH-required roles (§4.6.CP-029) are declared MUST but never given concrete content in this spec — the obligation floats without a target. The skill-declaration surface (§4.11) is a real contract but leaves at least one unresolved coupling to handler-contract §4.11 that makes it closer to a sketch than a ratified handshake.

The verdict is **proceed with revisions**. The four-Kind decision and the expression-grammar adoption need stronger justification text (or demotion to defaults-with-open-questions); the verdict-envelope question and the "concrete default permission set" gap are implementer-blocking.

## Challenges (5 load-bearing items)

### Challenge 1 — Budget-as-fourth-Kind is asserted, not earned; the §4.1.CP-005 table shows it is NOT a peer of Gate/Hook/Guard

- **Challenge** — the spec elevates Budget from "a concern gates enforce" (components.md §6.9 treats budget enforcement as a standalone section, not a subtype of the Gate/Hook/Guard primitive) to a first-class fourth `Kind` with equal status in the registry. The §4.1.CP-005 table is the evidence AGAINST this promotion: Budget shares no row-structure with Gate/Hook/Guard. Its trigger is "dispatch attempt + per-chunk accrual" (not transition/event/edge), its evaluator input is "counter state" (not a candidate transition, event, or edge set), its boundary rule is "mechanism only" (like Guard, unlike Gate/Hook), and its outcome actions `{admit, warn, deny}` are a fresh enum unrelated to the other three. Appendix A.3 concedes this ("Budget's trigger is dispatch + per-chunk accrual, not transition attempt") and defends the decision as "the unification is one registry, one lifecycle, one set of common fields" — but the §6.1 record schema then splits into four disjoint `KindPayload` variants, so the "common fields" are just `name`, `kind`, `evaluator`, `axes`, `mode_tag` — the minimum any registered primitive would carry.

- **What the spec says** — CP-001 ("single typed record… parameterized by `kind`"); CP-005 (per-Kind table); A.3 Rationale ("Why Budget is a Kind rather than a Gate subtype"); INFORMATIVE block under CP-005 ("Budget is a distinct Kind rather than a sub-category of Gate because its trigger… do not fit a Gate's transition-attempt trigger. The unification holds at the primitive level (one struct, one registry, one lifecycle) without forcing Budget into a Gate's shape.")

- **Is the justification adequate?** — no. The rationale argues against "Budget is a Gate subtype" (correct — it isn't), but the binary choice is not "Gate subtype" vs. "fourth Kind." The third option the spec never considers is "Budget is its OWN primitive registered in the same registry" — which is what CP-001 already produces structurally. The word "Kind" implies a taxonomic peer; the actual design is "typed record with four disjoint payload variants." If §6.1 shows four variants with no shared semantics beyond `name` + `evaluator`, the unification IS just shared registration. Either:
  - (a) rename "Kind" to something less peer-implying (e.g., `ControlPointType` with a comment "these are disjoint concerns sharing only registration"), and the decision becomes honest; or
  - (b) acknowledge that Budget's trigger-family is different enough that it should live as a standalone `BudgetPolicy` primitive registered in the same registry, with the four `Kind` values reserved for the actually-peer Gate/Hook/Guard. The latter matches components.md §6's original three-primitive framing.
  - The current framing reads as "we found a word to cover the awkwardness rather than resolving the awkwardness." First-plausible-answer.

- **How load-bearing** — important. The error category here is not correctness (the spec is self-consistent) but taxonomic drift: subsystem specs consuming this will encode `switch kind { case Gate: ...; case Budget: ...; }` where Budget's branch has no structural similarity to the others. Future additions (e.g., a Throttle primitive — per-second dispatch cap, not a per-run budget) will face the question "is this a fifth Kind or a subtype of Budget?" and the answer will be arbitrary because the Kind concept is already arbitrary. Closing the door now is cheaper than defending it later.

### Challenge 2 — `expr-lang/expr` adoption rejects alternatives that were never examined; cel-go is the missing comparator

- **Challenge** — CP-034 adopts `expr-lang/expr` as the normative grammar. The rationale (A.3 "Why `expr-lang/expr` and not a custom grammar") contrasts it against (a) "a custom grammar" and (b) "a general-purpose language (Starlark, Lua)" — both of which are obvious strawmen. The serious alternative is **cel-go** (Google's Common Expression Language), which is the industry standard for policy-expression grammars in Go (used in Kubernetes admission webhooks, OPA-adjacent systems, Envoy, gRPC transcoding). cel-go shares every property the rationale cites for `expr-lang/expr`: Go-native, maintained, published safety model (no I/O, no unbounded loops, bounded cost), type-checking at compile time, side-effect-free by construction. The rationale never mentions CEL by name.

- **What the spec says** — CP-034 ("The policy-expression grammar MUST be the grammar defined by `expr-lang/expr`"); INFORMATIVE callout ("maintained Go-native expression language with a known grammar and a published safety model"); A.3 rationale ("Why `expr-lang/expr` and not a custom grammar… Embedding a general-purpose language (Starlark, Lua) overshoots…"); OQ-CP-003 (defers version-pin policy but not the grammar choice).

- **Is the justification adequate?** — no. The rationale is written to defeat weak alternatives. The difference between `expr-lang/expr` and cel-go is not in the published-safety-model axis (both are safe) but in: (i) ecosystem maturity — CEL has a formal specification, cel-go is Google-maintained, and the grammar is used across cloud-native infra the harmonik operator likely already knows; (ii) typed environment — both offer type checking, but CEL's type system is more structural (protobuf-informed) and is a better fit for validating expressions against the Run/Outcome/Event record types at workflow-ingest time per CP-035; (iii) cross-tooling — CEL has language bindings beyond Go (useful if harmonik ever needs to let an IDE extension validate policy expressions); (iv) expressiveness constraints — `expr-lang/expr` is slightly more permissive (e.g., pipeline operators, built-in functions) in ways that enlarge the "cognition-creep-through-clever-expressions" surface the spec rationale itself worries about. The right shape is a two-column table: expr-lang vs cel-go, criteria in rows. Picking expr-lang without engaging cel-go is first-plausible-answer.

- **Stronger alternative** — either (a) add a brief rationale sub-section that compares expr-lang/expr vs cel-go by explicit criteria (grammar safety, type system, ecosystem, learning curve, policy-expression idioms in the foundation) and defends the choice; or (b) demote CP-034 from "MUST use expr-lang/expr" to "MUST use a side-effect-free, type-checkable expression grammar; expr-lang/expr is the default" and add OQ-CP-003b asking the grammar question explicitly. The spec already has precedent for "default with an Open Question" (OQ-CP-003 on version pinning). Broadening it to cover the grammar choice costs one paragraph.

- **How load-bearing** — important, not blocking. The decision is recoverable (swapping grammars is mechanical given the §4.7 environment is fixed), but recovery costs a migration release per CP-038. Closing the door now without examining cel-go is the kind of choice that operators discover and question later.

### Challenge 3 — "Persisted verdict ensures replay-safety" assumes the INPUT envelope is also pinned; the spec never says what the envelope contains

- **Challenge** — CP-041 and CP-INV-003 claim that persisting a cognition-tagged evaluator's verdict makes replay safe. But replay-safety requires that the verdict be **deterministic given the INPUTS** — if the inputs change between invocation and replay, reading the old verdict is stale, not safe. The spec names the verdict key as `{control_point_name, run_id, transition_id | event_id}` (CP-041), but a Gate evaluator's input per CP-005 is `{current state, candidate transition, outcome}` — which can differ between an original run and a replay if (a) the run's shared `context` map per CP-034 has been mutated between the invocation and the replay, (b) the prompt template per DelegationPath.prompt_template_ref has been updated, (c) the skill package provisioned at launch per §4.11 has a different version, or (d) the input_schema_ref has changed shape under N-1 evolution. The spec's replay rule says "read the persisted verdict" — but does not say "and verify the input envelope matches."

- **What the spec says** — CP-039 (delegation path named on the record); CP-040 ("verdict is persisted… BEFORE the transition advances"); CP-041 ("MUST NOT be re-invoked when a persisted verdict exists for the same `{control_point_name, run_id, transition_id | event_id}` key"); CP-042 ("persistence of the verdict is mechanism-tagged; the production of the verdict is cognition-tagged"); CP-INV-003.

- **Is the justification adequate?** — no. The mechanism does not ensure replay-safety; it ensures model non-re-invocation. These are different properties. Consider: run R₁ executes at t₀, Gate G fires, cognition produces verdict V with reasoning grounded in `context['budget_remaining'] = 500`. At t₁, reconciliation mutates `context['budget_remaining']` to `100` and triggers replay. CP-041 says replay reads V from git and proceeds — but V was produced against a context that no longer holds. The verdict is stale; reading it silently is a correctness bug, not a safety win. The ONLY way the persisted-verdict mechanism produces the claimed safety is if the envelope (inputs, prompt, skill versions, schema versions) is hash-pinned with the verdict and compared on replay. The spec says none of this.

- **Stronger alternative** — add CP-040b: "The persisted verdict record MUST include a `input_envelope_hash` field computed deterministically over the evaluator's input shape (context snapshot relevant to the expression, prompt_template_ref version, input_schema_ref version, and the skill package versions provisioned at the originating launch). On replay, the replayer MUST recompute the hash against the current envelope; a hash mismatch triggers a Cat 6 reconciliation verdict per CP-041 rather than a silent persisted-verdict read." Then CP-INV-003 reads "no re-invocation **with an unchanged input envelope**"; hash mismatch is the exit-door. Without this, the invariant is aspirational. The existing OQ-CP-004 raises a related unscoped-hook concern but misses the envelope question entirely.

- **How load-bearing** — blocking. This is the kind of correctness bug reconciliation will discover only after a subtle divergence produces an incorrect verdict, and the audit trail says "we read the persisted verdict per spec." The mechanism presented as a safety property is doing less work than claimed.

### Challenge 4 — "Concrete default permission sets" for MVH roles are declared MUST but never shipped; the spec has no teeth for CP-029

- **Challenge** — CP-029 ("For each MVH-required role (Planner, Builder, Reviewer…), the policy layer MUST supply a concrete default permission set (read-only vs. write to specific directories, tool whitelist, `default_skills`, spawn-source roles, Hooks that may modify behavior)") is a MUST requirement that names WHAT must exist but does not SHIP the content. A reader cannot answer "what tools does the default Builder role have?" from this spec. §6.2 gives the `PermissionSchema` shape; §6.3 shows the YAML template; neither shows filled-in default values.

- **What the spec says** — CP-028 (permission schema shape); CP-029 ("concrete default permission set… shipped at harmonik init"); CP-030 (deferred roles carry empty shells, with concrete empty values); CP-031 (default_skills MUST include Beads-CLI); §6.3 YAML shape (empty list templates).

- **Is the justification adequate?** — no. Conformance is unprovable: a Core MVH implementation that ships `allowed_tools: ["claude-code"]` for Builder satisfies CP-029 textually; so does one that ships `allowed_tools: ["claude-code", "bash", "git", "gh", "npm", "cargo", "write", "read"]`. Both "supply a concrete default permission set." The MUST has no observable violation surface. Either:
  - (a) this spec ships the MVH defaults in an appendix as a normative table (Planner: `tools=[read, plan, beads-cli]`, `writable_paths=["specs/**", ".kerf/**"]`; Builder: `tools=[read, write, bash, git, beads-cli]`, `writable_paths=["**"]`; Reviewer: `tools=[read, beads-cli]`, `writable_paths=[]`), making CP-029 checkable; or
  - (b) this spec says "the defaults are OUT OF SCOPE here and OWNED by [workflow-library.md §N]" and adds an Open Question OQ-CP-006 tracking the deferral; or
  - (c) the spec supplies illustrative defaults in §A.1 with an explicit non-normative marker and a pointer to where the authoritative defaults will land.
  - The current shape — "MUST exist, shape defined, content missing, no Open Question" — is the worst option because it claims normative coverage it does not provide.

- **How load-bearing** — blocking. The first downstream consumer (the S02 policy engine spec, or a bootstrap MVH policy YAML) will have to invent the defaults, and the invention will not be reviewable against this spec because this spec does not name them. Either supply them or defer them; the middle ground is dead weight.

### Challenge 5 — The §4.11 skill-declaration surface double-dispatches to handler-contract §4.11 without naming the handshake precisely

- **Challenge** — CP-049 through CP-052 declare the skill-declaration surface and say handler-contract §4.11 "consumes" it. But the handshake is under-specified in two ways: (i) the form of a `skill name` is never constrained (is it opaque? URL-shaped? a package coordinate with version? per §6.3 it is just "a skill name" as a string), and (ii) the resolution rule is under §4.11 of *this* spec ("resolves declared skills against available skill packages") but the actual resolution algorithm, fallback behavior, version-matching rule, and error taxonomy live in handler-contract.md §4.11 per the co-reference. Neither spec names WHO fails when a skill is declared at the node level but not installed on the machine — the node fails? the handler fails? the policy engine fails at registration? CP-050 says "the handler consumes this set… and fail-launches if resolution fails" which makes it handler-contract's problem, but CP-049 says `required_skills` "fail validation" at workflow-ingest time "unresolvable skill names fail validation" — which makes it the workflow validator's problem. Both cannot be true; one of them must be precise.

- **What the spec says** — CP-049 ("unresolvable skill names fail validation" at workflow-ingest time); CP-050 (effective set is union; handler consumes it and fail-launches on resolution failure); CP-051 (mechanism-tagged pipeline); §6.3 YAML shape (skills as string list).

- **Is the justification adequate?** — no. The spec commits to two failure modes (workflow-ingest-time validation, and handler launch-time fail-launch) without naming the invariant that distinguishes them. Candidate rules the spec does not state:
  - **Declaration validity** — the skill name must syntactically match an allowed shape (e.g., a lowercase-hyphenated identifier per the `agent_type` shape of architecture.md §1.6a, or a semver coordinate like `beads-cli@1.0.0`). The spec does not say.
  - **Registry-time resolution** — the S02 policy engine could maintain a "known skill names" list and fail registration for unknown names (like the Hook trigger list of CP-013). CP-049 implies this but does not name the registry.
  - **Launch-time resolution** — the handler searches `skill_search_paths[]` at agent launch and fails if the skill package is missing from the filesystem. CP-050 names this.
  - The honest shape is two layers: (1) a declarative validity check at workflow-ingest time (the skill name is syntactically valid and refers to a registered skill type) and (2) a provisioning check at agent launch (the skill package exists on disk). This spec conflates them under "unresolvable skill names fail validation."

- **Stronger alternative** — split CP-049 into two requirements: (a) CP-049a "A declared `required_skills` entry MUST match the skill-name shape defined in [handler-contract.md §4.11]; syntactic validation happens at workflow-ingest time" and (b) CP-049b "Resolution of a declared skill to a concrete skill package happens at agent launch per [handler-contract.md §4.11]; a missing package produces `ErrSkillProvisioningFailed` and routes through handler launch failure". Then the handshake is precise: this spec owns DECLARATION, handler-contract owns RESOLUTION. The co-reference is load-bearing in one direction (declaration → resolution) and does not round-trip.

- **How load-bearing** — important. The MVH will have a bootstrapping moment where the Beads-CLI skill is declared as a role default (CP-031) but not yet installed; the failure mode needs to be precisely one of "workflow won't validate" or "handler won't launch," not "it depends." Aspirational handshakes produce shipment-day surprises.

## Definitional gaps

Terms used heavily but not rigorously defined.

- **"Body" (as in CP-044 "identical body")** — the idempotency-by-name rule rests on comparing a registered ControlPoint's "body" against an incoming one. The spec never names which fields count as "body." `axes` is a derived classification; is it part of the body? `schema_version` — part of the body or metadata? The ambiguity produces a subtle failure mode: a policy author re-registers an identical-semantics Gate with a new `schema_version` bump, the comparison rejects it as divergent, startup fails. Add a definition: "The body of a ControlPoint is the tuple `(kind, trigger, evaluator, payload)`; `name`, `axes`, and `schema_version` are not part of the body."

- **"Tightest" (CP-022, CP-033)** — "the tightest applicable Budget wins" and "tightest wins" for freedom profiles. For integer-valued fields (limits, iterations) "tightest = smaller" is obvious; for enums (model tier, scope) it is not. CP-033 gestures at "the more restrictive enum" without declaring the ordering. A `model_tier` enum of `{haiku, sonnet, opus}` — which is tightest? The spec needs a per-enum ordering table OR a rule "for enums without a declared ordering, tightest-wins is unresolvable and both layers must be declared compatible."

- **"Attach point" vs "trigger"** — CP-005 uses "trigger" as the column header; CP-007 introduces "attach point" as a Gate-specific concept with four values. For Hooks and Guards, attach point is implicit (event subscription for Hooks, edge evaluation for Guards). Budget has no attach point. The spec inconsistently uses "trigger," "attach point," and "fires on" — three words for overlapping concepts. Either unify vocabulary or define each term relative to the others.

- **"Mechanism-tagged helper" (CP-012)** — Hook side-effects may "apply a declared state mutation (via a mechanism-tagged helper, never direct write)." No mechanism-tagged helper is defined in this spec or cited from another. What is the helper's surface? Who owns its contract? The requirement is load-bearing (it is the ONLY path for a Hook to mutate state) but points at nothing.

- **"External action" (§6.1.2 SideEffectKind)** — the `external-action` variant is listed but neither defined nor bounded. Is it a handler-owned effector (as CP-012 says)? Is it sandboxed? Can a Hook's external action call an LLM (and if so, is that a cognition-tagged Hook or still mechanism)? The side-effect taxonomy needs either a normative sub-section or a pointer to one.

- **"Idempotent by name" (CP-044)** — the word "idempotent" has a specific meaning in this corpus (per the idempotency axis). Registration is not literally idempotent — it is "safely re-applicable with identical body." The template §4.N+1 distinguishes idempotent from non-idempotent precisely. Using "idempotent" for the registration-surface guarantee conflates the registration property (same input → same registry state) with the axis tag. Prefer "re-registration-safe on identical body."

## MUST/SHOULD discipline

Six items where keyword choice hides a real requirement or over-claims.

1. **CP-009 Axes line is mis-tagged.** "Gate denial leaves the run in the source state… a `gate_denied` event is emitted" — the event emission is non-idempotent external I/O, so the Axes line `replay-safety=safe; idempotency=non-idempotent` is right. But **replay-safety=safe** contradicts the actual semantics: a gate denial IS a decision observable to improvement-loop audit. If the decision is replayed from the persisted verdict per §4.8, fine; if the denial path has NO persisted verdict (mechanism-tagged Gates don't produce one per CP-042), replay recomputes the expression — which IS safe if the expression is side-effect-free. The Axes are right; the reasoning is under-stated. Add one sentence: "Mechanism-tagged Gate denial is replay-safe because the expression is side-effect-free per CP-034; cognition-tagged Gate denial relies on persisted verdict per §4.8."

2. **CP-013 "MVH-baseline set is `on_agent_started`, …"** — this is MUST-shaped content in descriptive prose. Either the six-event set is normative (then the requirement should be "The following six lifecycle event types MUST be registered at MVH: …") or it is informative (then it belongs in `> INFORMATIVE:`). The current form lets an implementation skip `on_timeout` and still claim conformance.

3. **CP-025 "Default threshold is 0.8; operators MAY override"** — the "default" is a normative default (absent policy, the threshold is 0.8) but the requirement phrases it as a statement of fact. Compare against CP-022 which says "default 0.8; operator-configurable per §4.7 config precedence" — same information, also descriptive-not-normative. Pick one shape: either "unless overridden per §4.7, the threshold MUST be 0.8" or "the threshold is operator-configurable per §4.7 with a recommended value of 0.8."

4. **CP-027 "Reconciliation wall-clock budget is an outer bound"** — the entire requirement is a pointer to reconciliation.md §9.4a. It introduces no new normative content. This is §9 Cross-reference material, not §4 Normative. Either collapse into §9.3 Co-references or add a normative claim this requirement actually owns (e.g., "The outer-bound enforcement MUST NOT be bypassable by per-node or per-role Budget declarations; the outer Budget wins on any conflict" — which would be a real §4.5 obligation).

5. **CP-046 "Registry lookups are deterministic"** — the MUST list `{LookupByName, LookupByTrigger, LookupByAttachPoint}` names three methods but §6.1 `RECORD ControlPoint` doesn't declare a registry INTERFACE. The methods are asserted to exist. Either add an `INTERFACE Registry` block in §6 per template §6.1 showing the three methods and their contracts, or soften CP-046 to "the registry MUST expose deterministic lookups (methods defined in the S02 subsystem spec)."

6. **CP-015 "halt_on_failure = true" error mapping** — "Unhandled errors at the evaluator boundary map to `ErrTransient` or `ErrDeterministic` per [handler-contract.md §4.5] by the S05 dispatcher's wrapping rule." The OR is doing the work of a classifier that is never named. What decides `ErrTransient` vs `ErrDeterministic` for a Hook evaluator panic? Handler-contract §4.5 distinguishes them for handler errors, but Hook evaluators are not handlers. The wrapping rule is asserted to exist at the S05 boundary without being specified. This is a "the thing downstream does the work" deferral; make it explicit or name the classifier rule.

## Invariants that aren't really invariants

Applying template §5 selection test (invariant spans multiple subsystems; if it fits inside one §4 subsection, it is a requirement, not an invariant):

- **CP-INV-001 — Registry is the single source of truth.** HOLDS. Constrains S01, S02, S05 simultaneously.

- **CP-INV-002 — Effects are observable only through events.** HOLDS. Cross-subsystem by construction.

- **CP-INV-003 — Cognition-tagged evaluators never re-invoke the model during replay.** HOLDS — but see Challenge 3: the invariant is under-strength in its current form. Re-phrase to "no re-invocation with an unchanged input envelope" once CP-040b lands.

- **CP-INV-004 — Guards are mechanism-only.** FAILS the selection test. This is CP-020 verbatim. The rule is entirely inside §4.4; no other subsystem participates. Delete the §5 copy OR restate as "no subsystem may register a cognition-tagged Guard by any path" (which is what the registration surface of §4.9 already enforces, so still a duplication).

- **CP-INV-005 — Budget counter state does not leak.** FAILS the selection test. This is CP-026 verbatim. The rule is inside §4.5. Delete the §5 copy.

Two of five invariants (INV-004, INV-005) are §4-requirement restatements. Delete the §5 copies; the originals in §4 carry the normative weight.

## Hidden assumptions

Things the spec assumes that could turn out wrong.

1. **ControlPoint names are globally unique within the daemon without namespacing.** CP-002 says "unique `name` (String)" with no namespace. Policy documents from multiple sources (default, operator, workflow) merge per CP-037 precedence. A workflow author who names a Gate `deploy-approval` collides with the default policy's `deploy-approval`. The idempotency-by-name rule of CP-044 handles identical-body collisions, but different-body collisions fail startup. This is a usability trap: a workflow cannot safely introduce a Gate without knowing the whole default-policy name space. A `namespace/name` pattern (e.g., `workflow.task-pipeline.deploy-approval`) would solve this; the spec does not raise it.

2. **Policy documents are loaded in one startup phase, not dynamically.** CP-037 ("policy reloads are bound to the between-task invariant per [operator-nfr.md §7.3] — there are NO mid-run policy reloads") is the explicit rule, but the implicit assumption is that the entire policy corpus fits in memory at startup and is re-validated on every reload. No requirement bounds the policy-document size, the number of registered ControlPoints, or the registration time. A large installation with hundreds of workflows × tens of policies × dozens of ControlPoints per policy could produce startup times the operator feels. Not MVH-blocking, but a post-MVH trap.

3. **`expr-lang/expr` evaluation is safe against adversarial expressions.** CP-034 says "Expressions MUST be side-effect-free" and the INFORMATIVE block cites "bounded evaluation cost." But `expr-lang/expr` defaults are not hardened by default — there is a `expr.Timeout` option, a `expr.MaxNodes` option, but they are opt-in. The spec does not require the implementation to set them. An operator who writes a pathological expression (deeply nested map accesses, large slice comprehensions) could stall a Gate evaluation indefinitely. Add a CP-034b: "Expressions MUST be evaluated with a bounded cost ceiling configured by the implementation; the ceiling is a harmonik-level default, not a policy knob."

4. **Persisted verdicts are immutable once written.** CP-040 says verdicts are persisted to git; git commits are immutable. But CP-041 carves out a Cat 6 re-invocation path that "flags the verdict as stale" — which means either (i) the original verdict record must carry a `superseded-by` field pointing to the replacement (not in §6.1 `RECORD ControlPoint`), or (ii) the Cat 6 path writes a new verdict with the same key and git's `git show <path>` returns the newest (file-level overwrite on the task branch). The spec does not say which. Both are defensible; one must be named.

5. **The registry holds ControlPoint records by value, not by lazy reference.** CP-043 says "Go map keyed by `name`" — implying in-memory record storage. But policy documents may reference skill packages, prompt templates, schema definitions that are themselves loaded from disk. The spec treats ControlPoint registration as "parse YAML → construct record → register" in one pass (§7.1), but if a `prompt_template_ref` resolves lazily at invocation time, a prompt template deleted between registration and first use produces a runtime-discovered failure rather than a startup-discovered one. Lazy resolution is fine; undocumented lazy resolution is a correctness trap.

6. **Hook subsystem_priority integer ordering is globally comparable.** CP-014 (Hook ordering) sorts by "declaring-subsystem priority (explicit integer declared by the subsystem envelope)." Different subsystems declare priorities independently. If S05 declares priority 100 and S08 declares priority 100, the tiebreak falls to "declaration order" within the subsystem — but these Hooks are from different subsystems. The spec does not name the cross-subsystem tiebreak. Candidate: lexicographic over subsystem-ID. The rule must be named; silence invites non-determinism.

7. **The `event` field in the §6.4 policy environment is nil outside Hook context.** The policy environment schema declares `event: Event | nil`. If a mechanism-tagged Gate expression references `event.name`, the evaluator dereferences nil. The spec does not say whether this is a compile-time error (caught by expr-lang type-check at registration) or a runtime error (caught at invocation). expr-lang's type checker would catch this IF the type is declared as `Event | nil` and the expression dereferences; otherwise it is a runtime panic. Either is fine; the spec should name which.

8. **Gate "pre-selection" vs "post-selection" on transition attempt (CP-005, CP-006).** The CP-005 table says Gates fire on "transition attempt (pre-selection consumes outcome; post-selection consumes the chosen edge)." The attach-point enum in §6.1.1 names `edge-before-selection` and `edge-after-selection`. But CP-007 also allows `node-pre-entry` and `node-post-exit` — four attach points, two selection-points. When a Gate at `node-pre-entry` fires, is it before or after the cascade has selected the edge that led to that node? The semantics are scattered across CP-005, CP-006, and CP-007 and never consolidated. Add a table: attach point × what evaluator sees × who invokes.

9. **Freedom-profile "additive" vs "tightest-wins" (CP-032, CP-033).** CP-032 says "Freedom profiles are applied additively as agents traverse states." CP-033 says "the effective profile is the per-field intersection… tightest wins." These are contradictory adjectives: additive means union (more permissive), tightest-wins means intersection (more restrictive). The actual intent (per components.md §6.7) is "intersection/tightest-wins"; "additive" is the wrong word. Fix to avoid implementer confusion.

10. **Multi-document policy load order (OQ-CP-005 acknowledges this).** The default-if-unresolved says "cross-document references resolve after all documents are parsed but before registration; missing references fail startup." This is a reasonable default but the §7.1 pseudocode does NOT reflect it — the pseudocode parses and registers in one pass per document, which means cross-document references fail on the first-parsed document that references a not-yet-parsed target. Either update §7.1 pseudocode to match the OQ-CP-005 default (two-pass: parse all, then register) or close OQ-CP-005 with the single-pass alternative.

## Affirmations

Five decisions that hold up under pressure.

1. **Ownership split is clean and clearly cited (CP-010, CP-016, CP-021, CP-047).** S02 owns registry; S01 owns Gate/Guard invocation; S05 owns Hook dispatch. Each is named once, enforced once, and the "other two consult but do not own" wording makes the reader's job easy. The §4.10 consolidation repeats the rule for safety; this is the right place for it.

2. **Guard-is-mechanism-only (CP-020, CP-INV-004).** Rationale is airtight (cognition in Guard = ZFC violation at the selection layer + no persisted-verdict analog for a reordered-edge-list). The §A.3 paragraph justifying this is one of the strongest in the draft. INV-004 is redundant (see above) but the underlying rule is right.

3. **Separation of "declaration" from "resolution" for skills (CP-049–CP-052).** The three-artifact separation (DOT declares, YAML is the policy, handler provisions) is the right shape. The handshake needs sharpening per Challenge 5, but the directional split is correct.

4. **`schema_version` per policy document with N-1 readability (CP-038).** Consistent with execution-model.md EM-022 and operator-nfr.md §7.5. Doesn't try to over-engineer migration story; OQ-CP-001 defers the richer case honestly.

5. **Registration idempotency-by-name with divergent-body rejection (CP-044).** The "re-registering with identical body silently succeeds; divergent body fails startup" rule is exactly the right shape for a registration surface consumed by multiple subsystems. Prevents the "who wins?" question at merge time and produces an actionable error.

6. **DOT-references-YAML-by-name (CP-036).** The explicit "DOT attributes MUST NOT embed policy bodies inline" rule protects the three-artifact separation of architecture.md §1.9. This is a surface that subsystem specs would otherwise erode (a tempting optimization would be "inline small policies for single-use Gates"); closing the door early is the right move. The alternative — permitting inline bodies — would fragment the registry's single-source-of-truth role per CP-INV-001.

7. **Budget counter as event-only observable (CP-026, CP-INV-005).** Banning a `GetBudgetCounter()` read surface is architecturally correct — counter state is handler-internal per the goroutine-pinning rule of handler-contract.md §4.3, and cross-subsystem reads through shared memory would violate the centralized-controller principle of architecture.md §1.8. The event-bus-only channel forces backpressure and observability to be co-designed rather than bolted on. One of the cleanest structural calls in the spec.

## Cross-spec coupling observations

A few places where this spec's surface pokes neighboring specs in ways worth flagging to cross-spec-architect review:

- **Gate attach-points `node-pre-entry` / `node-post-exit` vs. edge cascade in execution-model.md §4.10.** The execution-model's EM-042 edge cascade consumes outcomes and selects edges; a Gate at `node-pre-entry` fires BEFORE the node's handler runs, which means BEFORE an outcome exists. CP-005 says Gates receive "the handler outcome that led to the attempt" — for a `node-pre-entry` Gate, that outcome is from the PREVIOUS node. This is probably intended but not stated; an implementer will reasonably expect `outcome = current-node's outcome` and be surprised.

- **Hook verdict persistence path `.harmonik/hooks/<run_id>/<hook_invocation_id>.json` (CP-040) vs. workspace-model.md §5.8.** The path lives on the run's task branch. But Hooks can fire on events emitted from sub-workflows (per execution-model EM-034, sub-workflow expansion is single-run), which means the parent run's task branch holds Hook verdicts fired from nested-node contexts. The `run_id` in the path is the parent run's ID; the `event_id` may be from a nested sub-workflow invocation. The path disambiguation works, but the sub-workflow correlation is implicit. Name the convention.

- **Budget wall-clock scope interaction with reconciliation (CP-027 + reconciliation §9.4a).** Reconciliation workflows carry a mandatory wall-clock Budget AND may themselves contain nodes that declare per-node Budgets. CP-027 says the reconciliation outer-bound "composes with any per-role or per-state Budgets." "Composes" is under-specified. If a per-node Budget of 600s exists inside a reconciliation run with a 300s outer bound, does the inner Budget denial produce a `budget_exhausted` or does the outer timeout fire first at 300s? Composition rule needs a worked example.

- **Skill declaration surface (CP-049-052) vs. handler-contract §4.11.** Already raised in Challenge 5. The co-reference arrow is unidirectional (this spec declares, handler-contract consumes), but the error taxonomy is split across both specs. A cross-spec-architect pass should verify the split is clean and the combined failure modes are a covering partition.

- **Event-emission WHEN (§6.5) vs. event-model.md §3.2 WHAT.** The spec correctly follows template §6.5 ("emitting spec is normative for the *when*; event-model is normative for the *shape*"). However, several events listed (`hook_verdict_persisted`, `skills_provisioned`, `guard_reordered`) may not yet exist in event-model.md's MVH taxonomy. Cross-check obligation: if any of these eleven events are absent from event-model, this spec's §6.5 is naming events that don't exist, and the co-reference is broken. Either event-model has them or a PR to add them must land with this spec, not after.
