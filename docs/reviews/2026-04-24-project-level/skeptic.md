# Skeptic Review — Project-Level Docs

## Verdict summary

Four of the five docs read as solid single-agent outputs with defensible defaults, but several load-bearing decisions are asserted rather than argued against their strongest alternatives. The two places the skeptic lens bites hardest are the **reviewer-agent-as-required-gate** (build-practices.md §Code review) and the **`faultpoint` compile-time-tag design** (testing.md §Crash-recovery) — both are presented as obviously-right when the counterfactual is actually competitive. A handful of smaller "one tool, one way" assertions (hand-written fakes, `go-arch-lint` over `depguard`, tiered CLAUDE.md) need sharper justification before they calcify.

## Challenges I want the user to see

### 1. Reviewer-agent as a *required* gate on every non-trivial PR is an unforced cost

- **Challenge**: Requiring a separate Claude Code session to post a structured verdict on every non-trivial PR is process the user explicitly rejected elsewhere, dressed up as automation.
- **What the doc says**: build-practices.md §Code review: *"Every non-trivial agent-authored PR passes two gates. 1. Reviewer-agent (required, mechanical). A separate Claude Code session invoked with the `review` skill reads the full diff, the touched specs, and the PR description. It posts a structured verdict comment: `APPROVE` / `REQUEST_CHANGES` / `BLOCK`."* And in the assumptions: *"I assumed the user wants an automated agent-review step on every non-trivial PR."*
- **Is the justification adequate?**: No. The justification is a bare assumption ("I assumed the user wants…"), and the listed criteria (spec alignment, ZFC classification, coverage, idiomatic Go, no new abstractions) overlap almost entirely with what the human reviewer is already doing in the 5–15 minutes the doc budgets. Solo-dev + agent-coded means the *author* is a Claude session and the *reviewer* is another Claude session reading the same specs — correlated failure modes. On a 1,000-LOC ceiling PR the human review budget is already the scarce resource; adding an asynchronous agent that must complete before merge converts every PR into a two-step workflow with an API-cost tail. And the `BLOCK` verdict in particular raises a governance question the doc doesn't address: if a Claude session can block a merge, what's the override path? (The doc doesn't say.)
- **What the stronger alternative would look like**: Make the reviewer-agent **on-demand, not required**. Human invokes it when (a) the diff is over some size threshold, (b) the diff touches cross-subsystem contracts, or (c) the human wants a second pair of eyes. Keep the `review` skill and its criteria list — drop the "required gate" framing. If you want a *required* automated gate that actually adds signal orthogonal to the author, it should be mechanical checks that an LLM is worse at (coverage delta, spec-section-cited presence, trailer-format, LOC budget) — not another LLM re-reading the diff.
- **How load-bearing**: **Blocking**. This is the kind of decision that hardens into ceremony once it's in the doc. The per-PR cost is recurring; cheaper to leave off than to remove later.

### 2. Hand-written fakes "because an agent can read them" bets against the thing agents are good at

- **Challenge**: The decision to ban `gomock`/`mockery` in favor of hand-written fakes rests on "hand-written fakes are ~20 lines and an agent can read them; generated mocks hide behavior" — but generated mocks are exactly the kind of repetitive code agents write well, and hand-written fakes rot silently when the interface grows.
- **What the doc says**: testing.md Decision 4: *"Mocking: hand-written fakes in `internal/<pkg>/faketest/`. No `gomock`, no `mockery`. Rationale: hand-written fakes are ~20 lines and an agent can read them; generated mocks hide behavior."* Assumption: *"Bets that ~5 subsystem interfaces is small enough for hand-written fakes to stay cheap. Revisit if interface count >15."*
- **Is the justification adequate?**: Partially. The "readability" argument is real but one-sided. The counter-argument the doc never engages: hand-written fakes drift. When an agent adds a method to the `Handler` interface, `gomock` regenerates and every test still compiles (with sensible defaults); with hand-written fakes, every `faketest/` in the tree needs a manual edit. Given the coverage aggression (95% on core, 100% of error returns), the number of interfaces that need fakes will not stay at "~5 subsystems"; it will include every seam the test layer exercises — easily 10–20 at MVH. The "revisit if >15" threshold is arbitrary and will be hit by the time MVH lands. Also: the ZFC tagging regime means some interfaces will have expectation-heavy tests (ordering, call count, argument matchers) — exactly what `gomock` does well and hand-fakes do badly.
- **What the stronger alternative would look like**: Hybrid — hand-written fakes for the 2–3 interfaces where behavior is the test (e.g., `EventBus`, where the fake's semantics are the contract), `mockery` for the rest. Or: hand-written fakes only where the fake is itself tested (a compliance suite over `Handler` fakes, which the twin-conformance strategy implies anyway). The current "no generators ever" rule is absolutist where a calibrated rule would be cheap.
- **How load-bearing**: **Important, not blocking**. The cost is paid in test-churn velocity; the signal that this was wrong will be visible in PR diffs that touch 5+ `faketest/` files at once.

### 3. `faultpoint` as a compile-tagged package is over-engineered for MVH crash testing

- **Challenge**: A dedicated package with named injection sites behind a `harmonik_fault` build tag is a lot of machinery for what the doc justifies with "production safety" — when the simpler alternative (runtime env-var flag, no-op in release) is cheaper and equally safe.
- **What the doc says**: testing.md §Crash-recovery: *"Fault injection is implemented by a `faultpoint` package: named injection sites compiled into the daemon behind a `harmonik_fault` build tag. In production builds, all calls are no-ops (dead-code-eliminated). In `//go:build crash` builds, `faultpoint.Arm(…)` causes the next hit of that named site to `syscall.Kill(os.Getpid(), SIGKILL)`."* Assumption: *"Build-tag chosen for production safety (zero overhead, impossible to arm by accident). Costs a second `go build` in CI."*
- **Is the justification adequate?**: Not really. "Impossible to arm by accident" assumes the attack model is an accidentally-armed fault in production — which for a solo-dev MVH daemon is nearly zero-probability. The actual cost is real: two build targets in CI, two sets of artifacts, a package whose API has to evolve in lockstep with wherever faults are injected, and a named-site list that becomes an ad-hoc spec ("what can we break?") the crash tests reference. The doc lists seven starter sites; in practice crash testing finds new sites constantly, and every new site is a cross-cutting edit. Compare to: a single `harmonik.FaultSite("mid-commit-write")` call that reads `os.Getenv("HARMONIK_FAULT")` at runtime and is a one-line no-op when unset. Same test ergonomics, half the build surface, no build-tag matrix.
- **What the stronger alternative would look like**: Runtime faultpoint with env-var arming; gate production builds by refusing to start if `HARMONIK_FAULT` is set and the binary isn't a `-tags=dev` build (single tag, one direction). Or: drop the named-site registry and inject faults via `context.Context` values that only test harnesses construct — zero production surface, zero new packages. Either is a smaller footprint for the same crash-testing power.
- **How load-bearing**: **Important**. The faultpoint design gets touched every time crash testing expands, which is often. Wrong shape here compounds.

### 4. "go-arch-lint" chosen over depguard on a one-maintainer tool; the fallback is named but not load-tested

- **Challenge**: The doc picks a third-party tool with one maintainer over a bundled-with-golangci-lint alternative on ergonomic grounds, then flags the bus-factor as a footnote — but the migration path ("drop in depguard") is asserted, not demonstrated, and the config styles don't actually map 1:1.
- **What the doc says**: subsystem-organization.md §Dependency layering: *"`depguard` (golangci-lint) works but its config is per-rule allow/deny lists scattered under `linters-settings`; architecture-as-component-graph is not its native model. `go-arch-lint` models a component graph…"* Assumption ⚑6: *"`go-arch-lint` is a third-party tool with one maintainer — pinning a specific version in `go.mod`/`tools.go` is prudent. If the tool goes unmaintained, `depguard` (bundled with `golangci-lint`) is a drop-in fallback."*
- **Is the justification adequate?**: Partially. The ergonomic argument is true — component graphs read better than per-file allow/deny. But `depguard` is *already* enabled in `quality-checks.md` (linter list) with a subsystem-boundaries rule; the two tools will co-exist, and the one-maintainer tool will be the source of truth for the rule that actually fires. Bus-factor aside, the more material risk is that `go-arch-lint` falls behind Go-version support or breaks on a module-graph change at exactly the wrong moment (MVH merge freeze). "Drop-in fallback" understates the work: translating 15 components with bidirectional dep rules into `depguard`'s linter-settings YAML is a 1–2 hour port under calm conditions, much more if the fallback is needed urgently.
- **What the stronger alternative would look like**: Write the architecture rules in *both* `go-arch-lint` (primary, for the nice error messages) and `depguard` (shadow, for portability and to keep the fallback warm). CI runs both; if they disagree, fail. One-time cost is the translation; ongoing cost is low because architecture changes are rare. Or: skip `go-arch-lint` entirely and make the `depguard` YAML less painful with a small generator from a component-graph DSL that lives in the repo. The current setup concentrates bus-factor risk on the hottest rule in the project.
- **How load-bearing**: **Nice-to-have, trending important**. If `go-arch-lint` stays maintained, this doesn't matter; if it doesn't, the rework lands during a crunch.

### 5. "Agents MUST wrap errors at subsystem boundaries" — where is the enforcement mechanism?

- **Challenge**: The error-handling convention says wrap with `%w` *at subsystem boundaries only* (not within), but there's no linter or mechanism that knows what a subsystem boundary is — so this devolves to reviewer vigilance, which is exactly the failure mode the rest of the doc is trying to mechanize away.
- **What the doc says**: quality-checks.md §Error handling: *"Wrap with `%w` at subsystem boundaries (crossing S01..S09), unwrap with `errors.Is`/`errors.As`. `errorlint` enforces both. Do NOT wrap within a subsystem."* And the deliberate omission of `wrapcheck`: *"No `wrapcheck` — Omitted deliberately; forces wrapping at every package boundary, conflicting with 'wrap only at subsystem boundaries.' `errorlint` + review cover the real cases."*
- **Is the justification adequate?**: No. `errorlint` checks that if you use `fmt.Errorf` with an error, you use `%w` — it does **not** enforce *where* wrapping should happen. It cannot; it has no concept of "subsystem." The doc admits as much by saying "errorlint + review cover the real cases" — which means the enforcement mechanism for a cross-cutting rule is "human eyes." Given the agent-coded context, this is exactly the kind of convention agents will violate invisibly: they'll wrap at package boundaries (the stdlib-ish pattern they've seen), not at S01..S09 boundaries (a harmonik-specific concept). And the review gate can't scale; even a reviewer-agent would need to import the subsystem graph to check this.
- **What the stronger alternative would look like**: A small custom analyzer (you have one planned already: "Custom analyzer for four-axis determinism tags" in the deferred list — same mechanism fits here). Feed it the `.go-arch-lint.yml` component graph; it flags any `fmt.Errorf("%w", err)` whose call site and return destination are in the same component, and any cross-component error return that *isn't* wrapped. Until that lands, be honest in the doc: downgrade "MUST wrap at boundaries" to "SHOULD wrap at boundaries; reviewer checks." The gap between asserted rule and enforcement mechanism is the exact anti-pattern the quality-checks doc opens by rejecting ("an agent must not be able to land low-quality or rule-violating code that CI lets through").
- **How load-bearing**: **Important**. This is the archetype of the bigger issue — there are several "agents MUST" rules in agent-configuration.md and build-practices.md (never amend, never force-push main, never `--no-verify`, never bundle unrelated changes) with no enforcement beyond norm. Each one is fine in isolation; collectively they are the seam where an agent-coded project quietly drifts from its spec.

## Affirmations

- **`internal/core` as a leaf package with an empty `mayDependOn`** — this is the right shape and will pay for itself the first time someone tries to add a convenience import from a subsystem.
- **Library allowlist enforced by a custom `forbid-import` CI check** — correct level of mechanism for the problem (stylistic drift in an agent-coded repo); cheap to maintain.
- **Default `go test ./...` runs unit+property only, tiers gated by build tags** — keeps the inner loop fast and the crash/integration cost visible only when you opt in. Sensible split.
- **`//go:build crash` fast subset (~1 min) vs full nightly** — pragmatic; avoids the common failure mode of crash suites that get disabled because they blow the CI budget.
- **Tiered CLAUDE.md with a 120-line cap on the root file** — the discipline is right. The root-file sprawl is the single most common failure mode of agent-config in real projects, and the cap is load-bearing.
- **"`harmonik-twin-*` as separate binaries" not in-process mocks** — aligns with the user's already-stated preference; the doc honors it without relitigating.

## Hidden assumptions

- **PRs will actually be ~400 LOC of Go**. Agent-coded PRs in practice trend 2–3x larger, especially with kerf-produced spec drafts + implementation bundled. The 1,000 LOC ceiling is flagged in the doc but the 400 LOC *target* is the number the reviewer-agent and human review are budgeted against; if actual median is 800–1200, every downstream time estimate is wrong.
- **"Subsystem count stays at ~10."** The `go-arch-lint` config, the depguard rules, the fakes-per-interface argument, and the coverage-class table all implicitly assume a stable 10-component structure. If MVH implementation finds natural splits (e.g., `orchestrator/runner` vs `orchestrator/router` graduating to peer subsystems), every one of those configs needs churn.
- **`claude-code` wire format is stable enough that recorded fixtures are useful**. Tier-C nightly fixture-refresh assumes drift is slow and detectable. If Anthropic ships a format change mid-sprint, the nightly PR will be the canary — but merging it will be non-mechanical every time, and the doc doesn't say who owns that.
- **The `agent-config-reviewer` skill will itself stay current.** The doc acknowledges this ("A skill reviewing the skills/rules config is recursive; the skill itself is the thing most likely to rot") and then doesn't actually propose a mechanism for keeping it fresh. If this skill goes stale, the Tier-2 cadence produces plausible-looking but wrong diffs that the main agent applies by default.
- **Branch protection + required CI status checks = agents cannot smuggle code past quality gates.** True mechanically for `main`, but agent workflows commit directly on `run/*` branches through the workflow engine (no PR), and the engine-driven merge to `harmonik/integration` bypasses branch protection entirely. The "CI is source of truth for mergeability" invariant applies only at the `integration → main` step, which is a smaller window than the doc's framing suggests. Worth naming explicitly before the workflow engine lands and this becomes a surprise.
- **Solo-dev stays solo through MVH.** Several choices (no approval requirement on branch protection, no release branches, memory as user-scoped, `gofumpt` over `gofmt`) are correct for one person and actively wrong for two. The promotion path is listed for each individually but not as a coherent "day-one-of-second-contributor" checklist — which is the moment these will all bite at once.
