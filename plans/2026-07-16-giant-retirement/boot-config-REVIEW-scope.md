# Adversarial scope/altitude review — `boot-config` giant-retirement design

> Read-only reviewer, 2026-07-16, branch `phase1-session-restart-substrate`. Lens: SCOPE, ALTITUDE, HONESTY, METHOD-FIT. Every source/config claim below re-verified against live tree.

## Verdict: **APPROVE-WITH-CHANGES**

The design is honest, well-sourced, and correctly method-fit; its own §7 refuses to claim the giant is "gone." Two load-bearing corrections must land before implementation (F1, F2), plus one scope decision the design defers but should not (F3). None are BLOCK-class — the seam is real and the sequencing is sound — but F1 in particular inverts the lint-gate math the design relies on, so it cannot ship as written.

## The crux: does this actually retire the giant?

**Partially — it retires the *function*, not the *mass*.** The single 1617-line `startWithHooks` genuinely collapses to a ~70-90-line linear phase sequence, and its branch concentration (137 `if` / cyclop) genuinely drops to ~10-14. That is a real, defensible win at the function-identity and complexity level. But the ~1450 lines of effectful wiring do not disappear — they relocate into **four new helpers of 180-300 lines each** (B3 `wireBusAndSubscribers` ~180, B4 `runStartupReconcile` ~280, B5 `wireSocketListener` ~250, B6 `launchWorkLoop` ~300), every one of them above the funlen-100 ceiling. So the honest summary is: **one 1617-line giant → one ~80-line shell + four mid-giants.** The design says this out loud (§7, "Do NOT claim the giant is gone"), which is to its credit. The failure is not dishonesty; it is that the *lint consequence* of that relocation is reasoned backwards (F1).

## Findings

### F1 — [load-bearing correction] The "grandfathered new helper" claim is mechanically wrong; the extraction *adds* funlen violations to the gate that actually runs at merge.
Evidence: the merge gate is `golangci-lint run --new-from-rev=HEAD~1` (Makefile:256, check-fast) / `--new-from-rev=origin/main` (Makefile:277, check-short/CI). The **full** `golangci-lint run` is explicitly *release-only, not a merge gate* and "fails on ~5666 pre-existing legacy issues" (Makefile:377-381). `--new-from-rev` grandfathers *pre-existing, unchanged* code; a **freshly extracted 180-300-line function has a brand-new func-decl line — it is exactly the "new" the ratchet is built to flag.** Therefore §7 option 1 ("accept them as new grandfathered functions … matches how `newWorkLoopDeps` is treated today") is false: `newWorkLoopDeps` is grandfathered *because it predates origin/main*; a new extraction inherits no such status. Net effect under the real gate: the shell may pass (its func-decl line is unchanged, so the ratchet's complexity anchor may not re-fire — the §1 ⚑ ambiguity), while B3-B6 **introduce four new funlen failures.** The refactor can make the merge-gate lint *worse*, not clean.
Required change: (a) stop measuring against "a full golangci-lint run" (§7, §0) — that bar is release-only and already red on 5666 issues; measure against `--new-from-rev`. (b) For each long phase helper, commit to either §7 **option 2 (sub-split under 100)** or an explicit `//nolint:funlen` with rationale — and state which, per helper, in the B3-B6 scope. The design's stated "no `//nolint` for funlen/cyclop/gocognit" goal (§7, §0) is only defensible for the shell; it is unattainable for the helpers without sub-splitting. This choice *is* the true scope of B3-B6 and cannot be left to "reviewer's call."

### F2 — [scope/altitude] The new-subsystem justification is thin, and the design asks the wrong question about it.
The design admits `bootconfig` shaves only ~35 lines (§0, B1: 1617→~1580) and that the shrink lives in B2-B6. It defends *why not bigger* (§2) but never defends *why a package at all*. The stated win — "config validation stops requiring a full `daemon.Start` boot; becomes pure table tests" (§6) — **does not require a separate Go package.** Pure functions in `package daemon` (a `bootconfig.go` file) are table-testable directly without booting anything. The *only* thing the package boundary buys over a daemon-internal pure file is the depguard import-isolation row — for four trivial string/`[]string`/`core.WorkflowMode` functions (~120 lines incl. tests). Measured against how `policy`/`orchestrator`/`hook` were justified (substantial pure-decision *domains* behind a hard edge), `bootconfig` is the smallest possible subsystem and reads closer to ceremony. It is *defensible* (the hard edge does prevent future drift of config-resolution into `queue`/`branching`, and it matches the overhaul's import-isolation thesis), which is why this is APPROVE-WITH-CHANGES not BLOCK — but the design must make the **package-vs-daemon-internal-pure-file** decision explicitly, with the depguard-drift argument as the stated reason, rather than assuming the package and asking the operator only to add the matrix row (open q1). As written, open q1 smuggles the real question ("is this a subsystem?") past the reviewer as a bookkeeping step ("add the row").

### F3 — [scope, must-decide] Open q3 (milestone bar) is the operator's real decision and B1 must not ship mislabeled.
B1 alone shaves 35 lines and does **not** retire the giant (design says so). Shipping B1 under the banner "giant retirement #1" would be a mislabel — B1 is "extract the config-resolution seam," full stop. Either the milestone is **B1-B6** (the only sequence that actually retires the function), or B1 ships under an honest name and the giant retirement is explicitly deferred to B2-B6. This is not an open question to hand downstream; it is a naming-honesty gate on the milestone. Recommend: milestone = B1-B6; B1 alone is not "giant retirement."

### F4 — [honesty: positive] The E/D/H seam map and the value split are honest and correctly scoped.
§1's classification (~1450 of 1617 lines are effectful composition-root wiring; only P2a/P2b/P2c ~35 lines are pure decision) is verified accurate against source (P1 pidfile at 784-808, P2a workflow-mode reject at 820-825, P2b branching merge at 837-848, all confirmed). §2's refusal to re-home `workspace.ValidateConflictResolutionAttemptCap` and to keep all I/O daemon-side is the correct envelope discipline. The depguard per-function proof (§4) is sound: all four funcs are `$gostd + core`. This is the strongest part of the design and needs no change.

### F5 — [method-fit: positive] Sub-slice sequencing matches the program method and is independently reviewable.
B1-B6 are each behavior-preserving, individually green-verifiable, and reviewed in isolation — mirroring the M5 3A/3B/3C cadence (design → adversarial seam review → worktree → cherry-pick `-x` → re-verify → independent reviewer → trailer) that COORD c050/c052 record as landed. The §8 load-bearing-invariant catalogue (pidfile `defer` must stay in shell scope — the highest-severity trap; Subscribe-before-Seal EV-009; `sleepBootBackoff` after socket bind hk-uzvt9; orphan-sweep before `LoadQueueAtStartup` QM-002a) is thorough and correct; the `bootState`-struct threading recommendation is the right shared-state mechanism. No change required to method.

### F6 — [minor] Open q4 caller set is slightly overstated.
`grep -rn resolveTargetBranch internal/daemon/*.go` shows callers at `daemon.go:864`, `workloop.go:1180`, `export_test.go:509` (+ def at `workloop.go:1312`) — not the "reconciliation scheduler at daemon.go:1649, :2302" the design cites. Small and non-load-bearing (the caller set is tiny and all in-daemon, so `daemon` importing `bootconfig` is trivial), but the implementer should re-grep rather than trust the cited line numbers.

## On the four open questions the design surfaces
- q1 (new subsystem row): right topic, **wrong framing** — see F2. It asks "add the row," not "should this be a package."
- q2 (funlen on helpers): right topic, but the design's own preferred answer (option 1) rests on the false grandfather premise — see F1. This is the load-bearing one.
- q3 (milestone scope): correct and critical — see F3; recommend resolving to B1-B6, not deferring.
- q4 (resolveTargetBranch ownership): fine — see F6.
- **Missing:** an explicit "which lint bar must the shrunk code pass" question (`--new-from-rev` vs the release-only full run). The design assumes the full run is the target; it is not the merge gate. Fold this into q2/F1.

## Bottom line
Approve the *direction* — the seam is real, the method is right, the honesty about the 35-line subsystem is refreshing. Before implementing: fix the funlen-gate reasoning (F1) and decide, per long helper, sub-split-vs-nolint; justify the package over a daemon-internal file (F2); and set the milestone at B1-B6 so "giant retirement" is not claimed by B1 alone (F3).
