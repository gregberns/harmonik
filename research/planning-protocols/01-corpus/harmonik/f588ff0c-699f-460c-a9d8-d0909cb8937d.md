## What DTW is

**DTW = Distributed Durable Workflow.** The canonical example is [Temporal](https://temporal.io) (and its predecessors: Uber's Cadence, AWS Step Functions, Microsoft's Durable Tasks, the newer Restate and DBOS projects).

The defining properties:

1. **Durable execution.** A workflow survives process crashes, machine failures, deploys. If the worker running it dies mid-step, a different worker picks it up and it resumes from where it left off — not from a checkpoint "a few minutes ago," but from the exact point of failure.
2. **Deterministic replay.** How it achieves that: the workflow function is a pure driver whose history is replayed from an append-only event log. Every side-effectful operation (HTTP calls, database writes, time-based logic) is wrapped in an "activity" that's recorded once; on replay, the recorded result is returned without re-executing. The workflow function ends up at the same state by "fast-forwarding" through its own history.
3. **Long-running by design.** Workflows can run for months or years. State isn't held in a process — it's held in the workflow service. `sleep(30 days)` is a legal operation.
4. **Distributed.** Workers are horizontally scalable; the central service coordinates handoffs.

**What harmonik has instead:** JSONL events as source of truth + git-commit-per-node checkpoints. This gives *durability* (restart from disk) but not *DTW-style deterministic replay* (harmonik doesn't re-execute handler functions from the event log — it resumes from the nearest checkpoint, and handlers are external subprocesses so they can't be "replayed" in the DTW sense anyway).

**Why Q-R2 exists.** If harmonik workflows run for many hours and must resume precisely after crashes without redoing LLM calls, that's DTW territory and Temporal/Restate/DBOS are the right references. If "restart = resume-from-last-checkpoint-with-some-re-work" is acceptable, harmonik's current model suffices. The research-pass recommendation was: study Temporal's *model* (event log + replay rules) selectively — which harmonik has already done implicitly — but skip the full DTW machinery. For MVH, not needed.

### The core idea

**Git history is the source of truth for completion.** The queue is a cache on top of it. If those two ever disagree, git wins, because git is append-only and commit hashes are stable — nothing can rewrite what happened.

1. **Task pulled from queue.** Queue row marked "in-flight" (atomic SQL transaction).

4. **Verification runs** (tests, lints). If pass, commits may or may not be added.
5. **Merge back to main.** The merge commit is THE marker that the task is complete — it carries `Harmonik-Status: completed`.
6. **Queue row updated** to "completed."

### Your three scenarios, walked through

**(1) Code commits, task-list status — staying in sync.**

The fix is: don't trust the queue as authority. On restart, rebuild queue status from git log.

Algorithm on restart:
- For every queue row that says "in-flight" or "pending with a workspace": scan git log for commits bearing that run's `Harmonik-Run-ID` trailer.
- If a `merged` transition commit exists on the main branch → task is actually completed. Queue is corrected to "completed."
- If commits exist on a branch but no merge → task was in-flight; workspace state tells us the rest (leased, merge-pending, etc.).
- If no commits exist at all → task never started or died in setup. Return to pending.

The queue state is fully derivable from git + the (much smaller) set of "pending, never started" entries. No silent drift because git can't lie.

Your "monotonically increasing" hunch was right: git commits ARE the monotonic sequence. The trailers make them queryable. The algorithm above is about 20 lines of Go.

**(2) Machine dies (battery), restart.**

Concretely, tonight's research decided:
- Queue is a SQLite file (`queue.db`). Binary boots, opens it. SQL transactions are crash-safe; the queue is already self-consistent.
- Event log is JSONL files (one per run). Binary re-opens them read-only to see recent events.
- Workspaces are git worktrees on disk. Binary scans them.
- Reconciliation (previous section) runs before the binary starts pulling new work.

Target restart time: under 15 seconds at 100k events, under 30s at a million. If it's more than 300 seconds, restart fails loudly and tells you to use `harmonik recover --from-checkpoint <id>`.

The restart is deterministic — it doesn't re-run any agent; it just reads disk and figures out where we were. This is NOT DTW (which would re-play function execution); it's much simpler.

**(3) Massive task set, crash mid-way.**

Your exact scenario: say 100 tasks queued, 50 done (commits on main), 1 in-flight (workspace exists, branch has 3 commits, no merge yet), 49 pending.

On restart:
- Queue says: 50 completed, 1 in-flight, 49 pending.
- Reconciler walks git log on main: 50 merge commits exist → queue correct for those.
- Reconciler looks at in-flight run's workspace: branch has commits, no merge → task 51 was interrupted.
- Queue status for task 51 changes from "in-flight" to something like "in-flight-interrupted"; workspace has `interrupt_cause` field set.
- **Operator decides for the interrupted one**: resume, rollback to last commit, or discard and re-queue. (This is the bit that isn't automatic — mid-task interruption needs judgment because an LLM call can't be "re-run identically.")
- Once the operator decides, binary resumes: pull task 52 next, continue.

The 49 pending tasks are untouched through all this — they're just queue entries that never left pending.

### The important caveat

This story assumes **single machine, single orchestrator process.** That's what MVH is.

The moment you want *two* orchestrators working the same queue from different machines, this model breaks. That's the scenario DTW solves. For now we're nowhere near it — and if we ever need it, we can migrate (JSONL events + git history are the kinds of things DTW systems consume comfortably).

## Where this leaves things

Nothing to decide right now — this is all consistent with what the research pass recommended. The restart-from-disk story is: SQLite queue + git history + JSONL event log + workspace state, reconciled deterministically on startup.

Is there a specific case you want to poke at? The "operator-must-decide for the interrupted one" step is the softest part of this — if you want the system to make automatic choices there instead, that's worth talking about.

## The one architectural choice I want to surface before we go further

Beads's bead `status` is coarse: `open / in_progress / closed / deferred / tombstone`. A harmonik task's *workflow* has many more states — planning, implementing, verifying, blocked-on-review, merge-pending, etc.

You have two ways to handle this mismatch:

**Option A — Harmonik owns workflow state; Beads sees only terminal transitions.**
Harmonik's JSONL event log + checkpoint trail holds fine-grained state (every node in every per-task workflow). Beads only sees: "bead claimed" (when harmonik starts work), "bead closed" (when the task workflow reaches its terminal merge), maybe "bead reopened" (after a failure classification). Everything else is invisible to Beads.

**Option B — Beads is the authoritative state store; harmonik writes every workflow transition into it.**
Every time a harmonik workflow node advances, we write to Beads. More visibility for other tools reading Beads; but the `blocked_issues_cache` thrashes on every micro-transition, and any other agent looking at Beads sees a firehose of "in_progress, in_progress, in_progress" sub-states.
