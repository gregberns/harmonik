## Analysis

### What the docs are vs. what specs need to be

The `./docs` are **design documents** — narrative, exploratory, full of rationale. They answer "why did we decide this?" Great for understanding the project, but problematic as implementation specs because:

1. **Normative content is buried in narrative.** An agent implementing `kerf finalize` has to extract the actual requirements from a user journey story, a CLI command description, a data model section, and resolved open questions — across 4 different files.

2. **Cross-cutting concerns are spread everywhere.** "Jigs" appear in 03, 04, 05, and 06. "Spec.yaml" is described in 03 and 05. An agent has to mentally merge these.

3. **Rationale and decisions are mixed with requirements.** "We chose X because Y" is useful history but noise when implementing. The agent needs to know *what* the system does, not *why* we chose it.

4. **No single source of truth per concept.** If a plan changes how finalization works, which doc is authoritative? The user journey in 02? The command spec in 04? The data model in 05?

### What specs should look like for this workflow

For the plan → spec → implementation loop, specs need to be:

- **Normative** — "the system does X", not "the idea is X"
- **Organized by domain** — one authoritative location per concept
- **Referenceable** — a plan says "change `specs/cli.md` section `kerf finalize`"
- **Machine-friendly** — structured enough that an agent can find what it needs without reading everything
- **Diffable** — when a plan modifies a spec, the change is localized and reviewable

### Proposed structure

  _index.md              # How specs work, format conventions, glossary

**9 spec files.** Each is substantial but focused. A typical plan touches 1-3 of them. The granularity is:
- Fine enough that "change how finalization works" means editing `cli.md` (the command) and maybe `data-model.md` (if the schema changes) — not rewriting a 300-line narrative doc

3. **How to create plans:** Format, what a plan must contain
4. **How to implement from specs:** Read the relevant specs, implement exactly what they say, verify consistency
5. **What to do when specs are ambiguous:** Flag it, don't guess
