## NTM (Named Tmux Manager) -- Research Summary

### What It Is

NTM is a Go binary that turns `tmux` into a control plane for running multiple AI coding agents (Claude Code, Codex, Gemini) in parallel. It handles session lifecycle, prompt dispatch, output monitoring, safety policy, and durable state -- all from the CLI or a REST/WebSocket API.

It is already installed locally at `/opt/homebrew/bin/ntm` and you have a `.ntm/` directory in the project root with logs, pids, and summaries from a previous session.

### How It Works

NTM creates a tmux session with labeled panes. Each pane runs an agent CLI (e.g., `claude` for Claude Code). A "user pane" (pane 0) is reserved for manual interaction. NTM tracks agent states (GENERATING, WAITING, THINKING, ERROR, STALLED) by monitoring pane output velocity and content.

### Installation

Already installed via Homebrew. Config lives at `~/.config/ntm/config.toml` (currently just `projects_base = "/Users/gb/github"`). Project-level config goes in `.ntm/`.

### Core Commands for Our Use Case

**1. Spawning agents with different missions:**

```bash
# Spawn 4 Claude Code agents for the secure-dev project
ntm spawn secure-dev --cc=4 --no-user

# With an initial prompt sent to all agents
ntm spawn secure-dev --cc=4 --prompt "Read AGENTS.md and report back"

# With staggered starts to avoid rate limiting
ntm spawn secure-dev --cc=4 --stagger-mode=smart

# With per-agent marching orders from a file
ntm spawn secure-dev --cc=4 --marching-orders orders.txt
```

**2. Sending different prompts to specific agents:**

```bash
# Send to all agents
ntm send secure-dev --all "Run go test ./... and report failures"

# Send to specific pane
ntm send secure-dev --pane=1 "Run every CLI command and report what breaks"
ntm send secure-dev --pane=2 "Execute user workflows end-to-end"
ntm send secure-dev --pane=3 "Test error handling edge cases"
ntm send secure-dev --pane=4 "Verify --json output on all commands"

# Send from a file
ntm send secure-dev --pane=1 --file prompts/cli-explorer.md

# With file context injection
ntm send secure-dev -c cmd/sd/main.go "Test all subcommands defined here"
```

**3. The `--marching-orders` flag** is the key feature for our scenario. It takes a file with pane-specific prompts in format `pane:N &lt;prompt&gt;`, allowing each agent to get a unique mission at spawn time.

**4. Pipelines for structured multi-agent workflows:**

Pipeline YAML files (`.ntm/pipelines/`) define multi-step workflows with:
- Steps with `agent`, `prompt`, `depends_on`, `when` conditionals
- `parallel` blocks for concurrent execution
- Variable substitution (`${vars.X}`, `${steps.X.output}`)
- Output parsing (JSON, regex, first_line)
- Retry logic with exponential backoff
- Loop constructs (for-each, while, times)

Example pipeline for our testing scenario:

### Limitations and Gotchas

1. **tmux-centric** -- everything goes through tmux. If tmux dies, state is lost (but checkpoints provide recovery).
2. **Rate limiting** -- multiple Claude Code agents will hit API rate limits. Use `--stagger-mode=smart` to mitigate. Still, 4+ Claude Code instances simultaneously may hit throttling.
3. **Agent output detection is heuristic** -- ntm detects agent states by monitoring output velocity and patterns. It can misclassify states, especially during slow generation.
4. **No direct "return value" from agents** -- agents write to the terminal, not to a structured channel. You collect results via `ntm extract`, `ntm summary`, `ntm grep`, or by having agents write to files.
5. **File conflicts** -- without `--worktrees`, multiple agents editing the same files will cause conflicts. For read-only testing (our use case), this is less of a concern.
6. **Pipeline `output_var` capture** relies on detecting when the agent finishes and parsing its terminal output. Complex or lengthy outputs may be truncated or misinterpreted.
7. **macOS and Linux only** -- no Windows support.
8. **Go 1.25+ claimed in their go.mod** -- the README says Go 1.25+ which does not exist yet (likely a mistake; they probably mean 1.22+ or use a toolchain directive).
9. **Optional integrations** (`br`, `bv`, Agent Mail, `cass`) enhance functionality but add complexity. For our use case, none of them are required.</result>
